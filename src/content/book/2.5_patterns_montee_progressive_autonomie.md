## Patterns de montée progressive en autonomie

On a maintenant le modèle de progression en trois phases, le HITL par design, et le gating à plusieurs niveaux. Mais il manque encore quelque chose : une réponse formelle à la question que tout le monde pose en réunion.

"Quand est-ce qu'on peut passer en phase suivante ?"

C'est une bonne question. Et si on n'a pas de réponse précise, la réponse par défaut sera soit l'impatience ("ça fait deux semaines, c'est bon non ?"), soit la peur ("on laisse comme ça encore un peu"). Les deux sont des mauvaises boussoles.

Ce qui manque, c'est un framework de gouvernance de l'autonomie. Pas une liste de bonnes pratiques — un système avec des critères mesurables, des seuils définis à l'avance, et une logique de décision qui s'applique sans discussion.

### Ce qu'on mesure avant de progresser

Le passage d'une phase à l'autre doit être justifié par des données. Trois métriques sont indispensables.

**La précision décisionnelle.** Sur les décisions prises par l'agent pendant la phase en cours, quel pourcentage était correct ? "Correct" est défini à l'avance par l'équipe métier — pas par le modèle. Pour un agent de support, correct signifie peut-être "la réponse a résolu le problème au premier contact". Pour un agent de facturation, correct signifie "le rapprochement était juste et le paiement approprié".

**Le taux de faux positifs.** Parmi les décisions gatées vers un humain, combien ont été approuvées sans modification ? Un taux de faux positifs élevé signifie que le gating est trop conservateur — et que l'agent est probablement prêt pour plus d'autonomie sur ces cas. Un taux qui baisse soudainement après avoir été stable est un signal d'alerte : quelque chose a changé dans le comportement de l'agent.

**L'absence d'erreur critique.** Aucun seuil de précision n'autorise le passage si l'agent a commis une erreur de catégorie 1 pendant la période d'observation — une décision irréversible incorrecte, une violation d'une règle métier fondamentale, une action avec impact client direct non planifié. Une seule erreur critique remet le compteur à zéro.

Ces trois métriques ensemble donnent une image fiable. Chacune seule est insuffisante.

### Les seuils de progression

Voici les critères que j'utilise comme base pour passer d'une phase à l'autre :

```python
PROGRESSION_CRITERIA = {
    "assisted_to_supervised": {
        "min_decisions": 100,        # Volume minimum observé
        "min_accuracy": 0.92,        # 92% de décisions correctes
        "max_false_positive_rate": 0.15,  # 15% max de faux positifs au gating
        "critical_errors_allowed": 0,     # Zéro tolérance sur erreurs critiques
        "observation_period_days": 30,    # Période d'observation minimum
    },
    "supervised_to_autonomous": {
        "min_decisions": 500,
        "min_accuracy": 0.97,
        "max_false_positive_rate": 0.05,
        "critical_errors_allowed": 0,
        "observation_period_days": 60,
    }
}

def can_progress_to_next_phase(metrics, current_phase) -> tuple[bool, str]:
    criteria = PROGRESSION_CRITERIA.get(current_phase)
    if not criteria:
        return False, "Phase inconnue ou terminale"

    if metrics.total_decisions < criteria["min_decisions"]:
        return False, f"Volume insuffisant : {metrics.total_decisions}/{criteria['min_decisions']}"

    if metrics.accuracy < criteria["min_accuracy"]:
        return False, f"Précision insuffisante : {metrics.accuracy:.2%}/{criteria['min_accuracy']:.2%}"

    if metrics.false_positive_rate > criteria["max_false_positive_rate"]:
        return False, f"Trop de faux positifs : {metrics.false_positive_rate:.2%}"

    if metrics.critical_errors > criteria["critical_errors_allowed"]:
        return False, f"Erreur(s) critique(s) détectée(s) : {metrics.critical_errors}"

    if metrics.observation_days < criteria["observation_period_days"]:
        return False, f"Période d'observation trop courte : {metrics.observation_days}/{criteria['observation_period_days']} jours"

    return True, "Tous les critères sont remplis"
```

Les seuils ci-dessus sont un point de départ, pas une vérité universelle. Un agent qui prend des décisions financières a besoin de critères plus stricts qu'un agent qui classe des emails. L'important, c'est que les seuils soient définis **avant** de commencer à collecter les données — pas après, quand on a envie de progresser.

### La rétrogradation : une fonctionnalité, pas un échec

Un système de progression sans rétrogradation n'est pas robuste. C'est une rampe sans filet.

La rétrogradation automatique, c'est le mécanisme qui ramène l'agent au niveau précédent si ses métriques se dégradent au-delà d'un seuil. Elle doit être implémentée dès le début, pas ajoutée en urgence après le premier incident.

**Exemple concret :**
Un agent de modération de contenu avait progressé en phase supervisée après deux mois de phase assistée solides. Précision à 94%, zéro erreur critique. Tout allait bien.

Trois semaines plus tard, son fournisseur LLM a déployé une mise à jour silencieuse du modèle. Rien dans les logs. Aucune notification. Mais la précision de l'agent sur les contenus ambigus est tombée de 94% à 81% en 72 heures.

Le monitoring l'a détecté. La rétrogradation automatique s'est déclenchée. L'agent est passé en phase assistée le temps qu'on analyse la cause — changement de comportement du modèle sur les nuances culturelles du corpus — et qu'on ajuste le prompt système.

Sans rétrogradation automatique, ces 72 heures auraient généré trois semaines de corrections manuelles.

```python
DEGRADATION_THRESHOLDS = {
    "supervised": {
        "min_accuracy": 0.88,        # En dessous → retour en phase assistée
        "max_critical_errors_7d": 1, # 1 erreur critique sur 7 jours → retour immédiat
    },
    "autonomous": {
        "min_accuracy": 0.94,        # En dessous → retour en phase supervisée
        "max_critical_errors_7d": 0, # Zéro tolérance en autonomie complète
    }
}

def check_for_degradation(metrics, current_phase) -> tuple[bool, str]:
    thresholds = DEGRADATION_THRESHOLDS.get(current_phase)
    if not thresholds:
        return False, "Phase sans seuil de rétrogradation"

    if metrics.accuracy_7d < thresholds["min_accuracy"]:
        return True, f"Dégradation de précision : {metrics.accuracy_7d:.2%}"

    if metrics.critical_errors_7d > thresholds["max_critical_errors_7d"]:
        return True, f"Erreur(s) critique(s) sur 7 jours : {metrics.critical_errors_7d}"

    return False, "Métriques stables"
```

### Ce que le monitoring doit capturer

Une montée en autonomie pilotée par les données exige un monitoring décisionnel — pas juste un monitoring des réponses.

La différence est importante. Un agent peut générer des réponses textuelles correctes et prendre de mauvaises décisions. Ce qu'on veut observer, c'est la **qualité du jugement** : quelle décision a été prise, dans quel contexte, avec quel résultat, sur quelle base.

[DIAGRAMME 2.4] Pipeline de monitoring décisionnel : action → niveau de gating → résultat → feedback → métriques

Le minimum viable : logger chaque décision avec son type, son niveau de gating, l'approbation ou le rejet si passage humain, et le résultat final observé. Sans ce logging, les métriques de progression sont impossibles à calculer et la rétrogradation automatique n'a rien sur quoi s'appuyer.

### Le livrable de la Partie 2

C'est ici que convergent les quatre sections de cette partie.

Le modèle Assisté → Supervisé → Autonome (2.2) donne la structure. Le HITL by design (2.3) place les checkpoints dans l'architecture. Le gating à trois niveaux (2.4) route chaque décision vers le bon contrôle. Et les patterns de progression formalisent la gouvernance de l'autonomie : quand progresser, quand rétrograder, et comment piloter la transition par les données.

Ensemble, ces quatre composants forment ce que j'appelle un **système de déploiement progressif** — l'alternative à l'agent autonome lancé le jour J et espéré le mois suivant.

Aucun de ces composants ne fonctionne seul. Le gating sans progression, c'est un agent bloqué indéfiniment. La progression sans rétrogradation, c'est une montée sans sécurité. Et tout ça sans monitoring décisionnel, c'est de la confiance aveugle.

Un agent n'est pas un prompt intelligent. C'est un système. Et comme tout système, il se pilote.
