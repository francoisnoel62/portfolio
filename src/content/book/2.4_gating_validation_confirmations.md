## Gating, validation, confirmations

La section précédente a posé la règle fondamentale : un checkpoint se justifie quand le coût d'une erreur dépasse le coût de l'interruption.

Maintenant, la question pratique : comment on structure ça dans le code ?

Parce que "mettre un humain dans la boucle" ne veut rien dire si tous les checkpoints se ressemblent. Un virement de 3€ vers un fournisseur récurrent ne mérite pas le même traitement qu'un licenciement automatisé. Pourtant, les deux peuvent théoriquement déclencher une "validation humaine".

Le gating bien conçu, c'est précisément ça : **router chaque décision vers le niveau de contrôle exact qu'elle mérite.** Ni plus, ni moins.

### Trois niveaux, pas deux

L'erreur la plus courante que j'ai vue : concevoir le gating comme un interrupteur binaire. Soit l'action passe en automatique, soit elle attend un humain.

C'est trop grossier.

Entre "automatique" et "humain", il y a une zone utile que la plupart des équipes ignorent : le gating semi-automatique. Un classifier qui inspecte l'action, applique des règles métier et prend une décision — sans impliquer un humain, mais sans non plus laisser l'agent faire ce qu'il veut.

[SCHÉMA 2.3] Les trois niveaux de gating : automatique → semi-automatique → humain

**Niveau 1 — Automatique.**
L'action passe sans validation.
Conditions : décision réversible, impact faible, périmètre connu, pattern déjà validé des centaines de fois.

Exemple : classer un ticket entrant dans une catégorie. Envoyer une confirmation de rendez-vous standardisée. Récupérer des données depuis une API en lecture seule.

**Niveau 2 — Semi-automatique.**
L'action est inspectée par un système de règles ou un classifier avant de passer.
Conditions : décision potentiellement sensible, mais critères objectivement évaluables.

Exemple : remboursement dont le montant est dans la plage habituelle mais le motif est inhabituel. Email sortant vers un nouveau domaine non encore vu. Modification d'un champ structuré dans la base de données.

**Niveau 3 — Humain.**
L'action attend une approbation explicite.
Conditions : décision irréversible, impact élevé, hors périmètre, ou confidence faible.

Exemple : suppression d'un compte client. Engagement contractuel. Tout ce qui déclenche un effet externe irréversible (email de licenciement, modification d'un contrat, paiement au-dessus d'un seuil).

### Implémenter le routeur de gating

Le cœur du système, c'est une fonction de routage. Elle prend une action en entrée et retourne le niveau de gating requis. Voici sa structure :

```python
from enum import Enum

class GatingLevel(Enum):
    AUTOMATIC = "automatic"
    SEMI_AUTO = "semi_automatic"
    HUMAN = "human"

def route_to_gating_level(action, context) -> GatingLevel:
    # Niveau 3 : humain obligatoire
    if action.is_irreversible:
        return GatingLevel.HUMAN
    if action.financial_impact > context.human_threshold:
        return GatingLevel.HUMAN
    if action.action_type in context.always_human_actions:
        return GatingLevel.HUMAN
    if action.confidence_score < context.min_confidence:
        return GatingLevel.HUMAN

    # Niveau 2 : inspection automatique
    if action.financial_impact > context.semi_auto_threshold:
        return GatingLevel.SEMI_AUTO
    if action.action_type in context.inspect_actions:
        return GatingLevel.SEMI_AUTO
    if action.touches_sensitive_data:
        return GatingLevel.SEMI_AUTO

    # Niveau 1 : passage direct
    return GatingLevel.AUTOMATIC

def execute_with_gating(agent, action, context):
    result = agent.plan(action)
    level = route_to_gating_level(result, context)

    if level == GatingLevel.AUTOMATIC:
        return execute(result)

    if level == GatingLevel.SEMI_AUTO:
        if not passes_automated_checks(result, context):
            return escalate_to_human(result, context, reason="failed_auto_check")
        return execute(result)

    # GatingLevel.HUMAN
    approved, feedback = request_human_approval(result)
    if feedback:
        log_feedback_for_improvement(feedback, result)
    if not approved:
        return handle_rejection(result, context)
    return execute(result)
```

Quelques détails qui comptent :

Le `semi_auto_threshold` et le `human_threshold` sont des paramètres de configuration — pas des constantes dans le code. Ils doivent pouvoir être ajustés sans redéploiement. Les seuils qui marchent en phase supervisée ne sont pas les mêmes que ceux de la phase autonome.

La fonction `passes_automated_checks` est le cœur du niveau 2. C'est là qu'on encode les règles métier : limites de montant par type de fournisseur, whitelist de domaines email, format attendu des données modifiées. Si quelque chose ne correspond pas au pattern attendu, on escalade directement au niveau humain.

### L'exemple de l'agent de facturation

**Exemple concret :**
J'ai travaillé avec une équipe finance qui avait un agent chargé de traiter les factures entrantes — vérification, rapprochement comptable, déclenchement du paiement.

Au départ, l'agent avait deux modes : soit il traitait tout seul, soit tout passait en revue humaine. Résultat : les comptables passaient 3 heures par jour à approuver des factures de fournisseurs récurrents avec des montants identiques aux mois précédents. Aucune valeur ajoutée, juste de la friction.

On a introduit trois niveaux de gating avec des critères précis :

- **Automatique** : fournisseur whitelisté, montant dans la tolérance de ±5% du mois précédent, format de facture valide.
- **Semi-automatique** : fournisseur connu mais montant atypique, ou nouveau format de facture — le classifier vérifie la cohérence avec les données contractuelles.
- **Humain** : nouveau fournisseur, montant au-dessus de 5 000€, facture avec mention légale inhabituelle.

En deux semaines, 73% des factures passaient en automatique. Les comptables ne voyaient plus que les cas qui méritaient vraiment leur attention. Le temps de traitement moyen est tombé de 2,3 jours à 4 heures.

### Ce qu'on gate, et ce qu'on ne gate pas

Une erreur symétrique à sous-gater : sur-gater.

Un système qui demande des confirmations à chaque étape finit par être ignoré. Les utilisateurs qui doivent approuver 40 actions par jour développent une approbation réflexe — ils cliquent "OK" sans lire. Le gating devient inutile, voire dangereux : il donne une fausse impression de contrôle.

La règle que j'applique : **si une action peut être approuvée sans être lue, elle ne devrait pas passer par le niveau humain.**

Ce n'est pas une règle contre la sécurité. C'est une règle pour que la sécurité soit réelle. Un humain qui valide 5 décisions par jour les lit. Un humain qui en valide 50 les approuve en automatique.

```python
# ❌ Gating indiscriminé — crée de la fatigue de validation
ALWAYS_VALIDATE = True  # Tout passe par un humain

# ✅ Gating calibré — l'attention humaine va là où elle compte
def should_involve_human(action, context):
    if action.is_irreversible:
        return True
    if action.financial_impact > context.human_threshold:
        return True
    # Éviter la fatigue de validation
    if action.is_routine and action.is_low_impact:
        return False
    return action.confidence_score < context.min_confidence
```

### Calibrer les seuils au fil du temps

Les seuils de gating ne sont pas des vérités absolues. Ils doivent évoluer avec la maturité de l'agent.

Au début de la phase supervisée, on gate large : mieux vaut trop contrôler que pas assez. Les faux positifs (actions inoffensives qui passent en revue humaine) sont acceptables — ils alimentent le diagnostic.

Avec le temps, on exploite les données de validation pour ajuster. Si une catégorie d'actions est approuvée à 98% sans modification depuis 3 mois, c'est un candidat pour passer en semi-automatique ou en automatique. Si une catégorie génère 30% de rejets humains, c'est un signal qu'il faut revoir le comportement de l'agent sur ce type d'action.

Le gating n'est pas une configuration statique. C'est une variable d'ajustement de la progression vers l'autonomie.
