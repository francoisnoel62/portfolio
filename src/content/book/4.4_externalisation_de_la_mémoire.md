# 4.4 Externalisation de la mémoire

La conversation est éphémère par nature. Le state est structuré et persistant par design.

On l'a établi en 4.3. Mais "persistant par design" ne signifie pas grand-chose si on n'a pas répondu à la question qui suit : persistant *où* ? Et comment ?

C'est là que la plupart des implémentations tombent dans un piège simple. On décide d'externaliser la mémoire — bonne décision — et on met tout dans une base de données. Une seule. Pratique, rapide à déployer, cohérent en apparence. Et profondément inadapté.

Parce que tout ce qu'un agent doit mémoriser n'est pas de la même nature. Les types de mémoire qu'on a identifiés en 4.2 — court terme, long terme — obéissent à des logiques d'accès radicalement différentes. Les stocker au même endroit, avec les mêmes mécanismes de requête, c'est comme ranger ses outils, ses vêtements et ses documents dans le même tiroir. Ça rentre. Mais retrouver quelque chose au bon moment devient un problème que l'architecture aurait dû résoudre dès le départ.

### Trois types de stores, trois logiques différentes

**Le premier type : la base relationnelle pour le state structuré.**

Le state — dates de réservation, statut d'une commande, décisions prises, données métier collectées — est structuré par définition. Il s'adapte parfaitement à une base relationnelle. PostgreSQL pour les déploiements sérieux, SQLite pour les contextes plus légers. Les requêtes sont précises et déterministes : `WHERE user_id = X AND status = 'pending'`. Les transactions garantissent l'intégrité même sous charge. La lecture est fiable.

C'est le bon endroit pour les informations dont on connaît la structure à l'avance. Celles qui ne changent pas de forme d'une session à l'autre. Celles qu'on interroge avec des critères explicites, pas avec une approximation sémantique.

**Le deuxième type : le vector store pour la mémoire sémantique.**

On a vu en section 3.4 ce que les embeddings vectoriels permettent pour le retrieval documentaire. La même logique s'applique à la mémoire long terme de l'agent. Les souvenirs sémantiques — résumés de sessions passées, faits appris sur l'utilisateur, préférences exprimées en langage naturel — ne se retrouvent pas avec une requête SQL. On ne sait pas à l'avance ce qu'on va chercher. On cherche par proximité : "Qu'est-ce que je sais sur cet utilisateur qui est pertinent pour cette requête ?"

C'est le rôle du vector store. Chroma, Pinecone, Weaviate, ou pgvector si on reste dans PostgreSQL. On stocke des embeddings des informations mémorisées et on les retrouve par similarité sémantique avec la requête en cours. La précision n'est pas binaire comme en SQL — c'est une question de degré de pertinence. Et c'est exactement ce qu'on veut pour une mémoire contextuelle.

**Le troisième type : le cache pour l'accès rapide.**

Redis ou un cache en mémoire pour ce qui doit être accessible immédiatement et fréquemment. Les données de la session active. Le profil de l'utilisateur courant qu'on va injecter à chaque appel. Les résultats intermédiaires qu'on ne veut pas recalculer à chaque requête de la même session.

Le cache n'est pas une mémoire permanente — c'est un accélérateur. Sa durée de vie est courte : quelques minutes à quelques heures, selon la durée typique d'une session. Passé ce délai, on retourne chercher en base. Ce TTL court est une fonctionnalité, pas une limitation : il garantit que les données de session ne s'accumulent pas indéfiniment.

### Pourquoi le mélange casse

**Exemple concret :**

J'ai accompagné une équipe qui déployait un agent de gestion de projet pour un groupe de douze personnes. L'agent devait maintenir le contexte de chaque projet : statut des tâches, décisions prises en réunion, préférences de l'équipe pour le format des comptes-rendus.

Au départ, tout allait dans PostgreSQL. Le statut des tâches : facile à requêter. Les décisions de réunion : stockées en JSON dans une colonne `text`. Les préférences de format : idem. Ça marchait. Jusqu'au moment où on a voulu retrouver des décisions passées pertinentes pour une nouvelle réunion.

"Qu'avons-nous décidé la dernière fois sur ce sujet ?" Cette question simple devenait un cauchemar. Une recherche SQL sur du texte libre, c'est une recherche par mots-clés. Si la question était formulée différemment, si le compte-rendu avait utilisé des synonymes, si le contexte thématique avait légèrement changé — on ratait des décisions importantes. L'agent donnait des réponses incomplètes sur la base d'une mémoire mal interrogée.

La solution était architecturale, pas technique. Déplacer les décisions de réunion vers un vector store. Le statut des tâches reste en base relationnelle — structuré, interrogeable précisément. Les décisions de réunion passent en embeddings — retrouvées par similarité, sans friction lexicale. Les données de session active vont en cache — chargées une fois, accessibles partout dans la session en cours.

Trois stores. Trois logiques. Une architecture qui correspond enfin aux besoins réels.

### La couche mémoire : une interface sur la complexité

Ce qui lie ces trois stores, c'est ce qu'on appelle le **memory layer** — la couche mémoire. Une abstraction qui cache la complexité derrière une interface simple : `remember()` pour stocker, `recall()` pour retrouver.

L'agent n'interagit pas directement avec les stores. Il passe par cette couche, qui décide où stocker selon le type d'information, et où chercher selon le type de requête. Le code de l'agent reste propre — il n'a pas à connaître les détails d'implémentation de chaque store.

```python
# ❌ Accès direct aux stores — logique dispersée, couplage fort
def handle_session(user_id: str, message: str) -> str:
    # L'agent gère lui-même trois stores différents — fragile
    user_state = postgres.query("SELECT * FROM states WHERE user_id = %s", user_id)
    past_context = vector_store.search(message, top_k=3)
    session_data = redis.get(f"session:{user_id}")

    response = llm.generate(
        context=build_context(user_state, past_context, session_data),
        message=message
    )

    # Persistance dispersée — chaque modification touche plusieurs stores
    postgres.update("UPDATE states SET last_seen = NOW() WHERE user_id = %s", user_id)
    vector_store.upsert(user_id, response, {"timestamp": now()})
    redis.set(f"session:{user_id}", session_data)
    return response


# ✅ Couche mémoire — interface unifiée, décisions centralisées
class MemoryLayer:
    def __init__(self, user_id: str):
        self.user_id = user_id
        self._db = PostgresStore()          # State structuré
        self._vectors = VectorStore()       # Mémoire sémantique
        self._cache = RedisCache(ttl=3600)  # Session active

    def remember(self, content: str, memory_type: str, metadata: dict = None) -> None:
        """Stocke une information dans le bon store selon son type."""
        if memory_type == "state":
            self._db.upsert(self.user_id, metadata or {})
        elif memory_type == "semantic":
            self._vectors.upsert(self.user_id, content, metadata or {})
        elif memory_type == "session":
            key = (metadata or {}).get("key", "default")
            self._cache.set(f"{self.user_id}:{key}", content)

    def recall(self, query: str, memory_type: str = "all") -> dict:
        """Retrouve les informations pertinentes depuis le bon store."""
        result = {}
        if memory_type in ("state", "all"):
            result["state"] = self._db.get(self.user_id)
        if memory_type in ("semantic", "all"):
            result["context"] = self._vectors.search(
                query, user_id=self.user_id, top_k=3
            )
        if memory_type in ("session", "all"):
            result["session"] = self._cache.get_all(self.user_id)
        return result


def handle_session(memory: MemoryLayer, message: str) -> str:
    # L'agent demande ce dont il a besoin — la couche mémoire gère le reste
    context = memory.recall(query=message)

    response = llm.generate(
        state=context.get("state"),
        past_context=context.get("context"),
        session=context.get("session"),
        message=message
    )

    # Mise à jour centralisée — une seule interface
    memory.remember(response, memory_type="semantic", metadata={"timestamp": now()})
    return response
```

[ARCHITECTURE 4.4] Memory Layer — trois stores, une interface : PostgreSQL (state structuré), Vector Store (mémoire sémantique), Redis (session active)

L'avantage de cette abstraction dépasse la propreté du code. Elle permet de changer un store sous-jacent sans toucher à la logique de l'agent. Elle rend la couche mémoire testable indépendamment du reste du système. Et elle documente explicitement les décisions architecturales : ce qui va où, et pourquoi.

### Ce que l'architecture change en production

Avec ce pattern en place, trois problèmes disparaissent.

La persistance inter-sessions fonctionne vraiment. L'utilisateur revient le lendemain — son state est rechargé depuis la base relationnelle, le contexte sémantique pertinent est retrouvé depuis le vector store, la session repart avec les bonnes informations. Pas d'amnésie. Pas de questions répétées.

La scalabilité est gérée par les bons outils. Le vector store monte en charge sur des millions d'embeddings sans dégradation des requêtes sémantiques. La base relationnelle gère les accès concurrents avec les garanties transactionnelles dont on a besoin. Le cache absorbe les pics de lecture sans solliciter les stores persistants. Chaque composant est dimensionné pour son usage réel.

Le débogage devient enfin possible. Quand un agent prend une mauvaise décision, on peut tracer précisément : quel state a été injecté, quel contexte sémantique a été retrouvé, quelles données de session étaient actives. La mémoire n'est plus une boîte noire — elle est observable, interrogeable, auditée.

C'est ça, l'externalisation de la mémoire bien faite. Pas un tiroir fourre-tout. Une architecture réfléchie où chaque type d'information va au bon endroit, où les accès sont optimisés pour chaque usage, et où la couche mémoire garantit que l'agent reçoit toujours ce dont il a besoin — sans avoir à savoir où ça vit.

Il reste une dernière question. On sait maintenant stocker et retrouver. Mais comment s'assurer que ce qu'on retrouve appartient bien à la bonne entité, au bon projet, au bon fil d'histoire ? C'est la question de l'identité et de la continuité. On l'aborde en 4.5.
