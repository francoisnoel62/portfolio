## Multi-tenant, RBAC et isolation

Jusqu'ici, on a raisonné comme s'il y avait un seul utilisateur. En production, un agent sert souvent plusieurs clients — plusieurs *tenants* — sur la même infrastructure.

Et là, une question banale devient critique : qu'est-ce qui garantit que l'agent du client A ne voit jamais les données du client B ?

Si la réponse est « le prompt lui dit de ne pas le faire », vous avez un problème. On l'a vu en Partie 7 : la sécurité ne repose pas sur le bon vouloir du modèle. L'isolation multi-tenant non plus.

### Isoler les données, jusqu'à la mémoire

Le premier cloisonnement est celui des données. Chaque donnée porte un `tenant_id`, et chaque accès est filtré dessus — pas par l'agent, par l'infrastructure.

Le piège spécifique aux agents, c'est la **mémoire** et le **retrieval**. On l'a construit en 4.5 : la mémoire long terme se partitionne par identité. En multi-tenant, c'est vital. Un vector store partagé sans filtre par tenant, et l'agent du client A peut « se souvenir » d'un document du client B — une fuite inter-tenant servie par le retrieval lui-même.

```python
# ❌ Retrieval sans filtre tenant : fuite inter-client par similarité
docs = vector_store.search(query, top_k=4)

# ✅ Le filtre tenant est appliqué AVANT la recherche, côté infrastructure
docs = vector_store.search(
    query, top_k=4,
    filter={"tenant_id": ctx.tenant_id},   # jamais l'agent ne choisit le tenant
)
```

Le `tenant_id` vient du contexte d'exécution authentifié, jamais de quelque chose que le modèle aurait pu produire ou qu'une donnée aurait pu injecter (rappel de la prompt injection, 7.2). C'est une frontière de confiance, et elle est dans le code.

### RBAC : l'agent agit au nom d'un acteur

Un agent n'a pas de droits propres. Il agit **au nom de** quelqu'un — un utilisateur, un service — avec les permissions de cet acteur, et pas plus.

C'est le RBAC (contrôle d'accès basé sur les rôles), et on l'a déjà croisé en 7.2 avec `actor.is_allowed(action)`. Ici on le systématise : chaque action que l'agent veut déclencher est vérifiée contre les droits de l'acteur réel, au moment de l'exécution.

```python
# ✅ L'agent propose ; les droits de l'ACTEUR (pas de l'agent) tranchent
def execute_for(actor, action, ctx):
    if action.tenant_id != ctx.tenant_id:
        return deny(action, "frontière de tenant")          # cloisonnement dur
    if not actor.has_permission(action.required_role):
        return deny(action, "permission insuffisante")      # RBAC
    return execute(action)
```

Conséquence importante : deux utilisateurs du même tenant peuvent obtenir des résultats différents du même agent, parce qu'ils n'ont pas les mêmes droits. C'est voulu. L'agent hérite des permissions de celui pour qui il agit — il ne les élargit jamais.

### Le voisin bruyant

Dernier risque du multi-tenant : un client qui consomme tout. Un tenant qui lance mille tâches sature les workers, fait exploser la facture de tokens, et dégrade le service de tous les autres. C'est le « noisy neighbor », et côté agents il prend une forme aiguë à cause du coût des LLM (la consommation non bornée, LLM10 de la Partie 7).

Les garde-fous :

- **Quotas par tenant** : nombre de tâches concurrentes, requêtes par minute.
- **Budget de tokens / coût par tenant** : un plafond, au-delà duquel on throttle ou on alerte.
- **Files isolées ou pondérées** : un tenant ne doit pas pouvoir monopoliser la file de jobs (9.2).

**Exemple concret :**
En contexte d'intégration multi-clients, un même agent servait plusieurs entreprises sur la même plateforme. Sans quota, un client a lancé un import massif un lundi matin : la file de jobs s'est remplie de ses tâches, et les agents des autres clients ont attendu des heures. Le correctif n'était pas un meilleur modèle — c'étaient des files pondérées par tenant et un plafond de concurrence par client. Personne ne pouvait plus, seul, dégrader le service de tous.

Une remarque qui relie cette section à la Partie 7 : une fuite inter-tenant n'est pas qu'un bug d'isolation, c'est un incident de sécurité et de conformité — les données d'un client exposées à un autre. On la traite donc avec le même principe que la sécurité : **deny par défaut.** L'accès n'est pas « autorisé sauf si interdit » mais « interdit sauf si explicitement autorisé pour ce tenant et cet acteur ». Sur un système multi-clients, c'est cette inversion — refuser par défaut — qui évite que la prochaine fonctionnalité ajoutée ouvre, sans qu'on s'en aperçoive, une porte entre deux clients.

### Le principe

L'isolation multi-tenant tient en une phrase : **le tenant et les permissions viennent du contexte authentifié, jamais du modèle ni des données.** Données filtrées par tenant, actions vérifiées par RBAC, ressources plafonnées par client. Trois cloisons, toutes dans l'infrastructure, aucune dans le prompt.

On sait maintenant faire tourner l'agent de façon fiable et isolée. Reste à le *voir* tourner — et à savoir quoi faire quand il dérape.
