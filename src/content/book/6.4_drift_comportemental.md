## Drift comportemental

On vient de poser la machinerie. Le `DecisionLog` capture chaque décision. Le dashboard regarde le raisonnement, la confidence, les résultats. On sait ce que l'agent décide et pourquoi.

C'est nécessaire. Ce n'est pas suffisant.

Il reste le problème du drift.

En Partie 2, j'ai introduit le drift comportemental comme "changement non désiré du comportement d'un agent après mise à jour ou changement de données". En pratique, ça ressemble à ça : l'agent glisse. Sans casser. Sans exception levée. Sans alerte qui se déclenche. Il continue à répondre, à décider, à agir — mais différemment de ce pour quoi il a été calibré.

C'est précisément ce qui le rend dangereux.

### Ce qui provoque la dérive

J'ai observé trois mécanismes distincts en production.

Le premier, c'est le changement de modèle. Ton provider met à jour silencieusement le LLM sous-jacent. Tu n'as rien changé dans ton code. Tu n'as rien changé dans tes prompts. Mais le comportement change — parfois légèrement, parfois significativement. Une nouvelle version peut être plus prudente, plus créative, ou plus littérale dans son interprétation des instructions. L'effet sur un agent de production met souvent des semaines à se manifester, parce que le changement s'opère sur la marge : les cas ordinaires restent identiques, les cas limites basculent.

Le deuxième, c'est le changement de données. Les données qui arrivent à l'agent évoluent. Les patterns de requêtes se transforment. Les saisonnalités changent. Un agent de modération calibré sur les données de janvier peut ne pas être prêt pour les contenus de décembre. Un agent de support voit émerger de nouvelles typologies de problèmes que ses exemples d'entraînement ne couvrent pas. Le contexte a changé — mais l'agent répond avec les mêmes règles d'il y a six mois.

Le troisième, c'est le changement d'usage. Les utilisateurs apprennent. Ils découvrent comment formuler leurs demandes pour obtenir ce qu'ils cherchent. Des formulations qui contournent les règles sans les violer explicitement. Des patterns qui activent des comportements que tu n'avais pas prévus. Ce n'est pas de la manipulation intentionnelle. C'est l'adaptation naturelle des utilisateurs à un outil. Mais son effet sur le comportement de l'agent est réel.

**Exemple concret :**
J'ai suivi un agent de pricing dans une plateforme e-commerce pendant plusieurs mois. En janvier, il calibrait correctement les remises selon les règles métier. En avril, l'équipe a remarqué une anomalie : l'agent accordait des remises systématiquement plus basses que les cibles — 7 à 12% sous les niveaux attendus. Aucune mise à jour n'avait été déployée côté code. Les données de pricing n'avaient pas changé. Mais le provider LLM avait mis à jour silencieusement son modèle en mars. La nouvelle version était légèrement plus "conservatrice" dans son interprétation des règles d'exception. Résultat : trois mois de sous-performance sur les campagnes promotionnelles. Invisible sans monitoring décisionnel.

### Détecter le drift avant qu'il fasse des dégâts

La baseline constituée en 6.2 n'est pas juste une référence historique. C'est le point de comparaison actif contre lequel on vérifie le comportement courant.

La détection du drift repose sur une idée simple : comparer les distributions des décisions sur des périodes glissantes. Si la distribution de la semaine en cours s'écarte significativement de la baseline — ou de la semaine précédente —, c'est un signal à investiguer.

```python
# ✅ Détection de drift par comparaison de distributions
from collections import Counter
from dataclasses import dataclass
from typing import List

@dataclass
class DriftReport:
    detected: bool
    signal: str
    baseline_distribution: dict
    current_distribution: dict
    divergence: float

def detect_decision_drift(
    baseline_decisions: List[str],
    current_decisions: List[str],
    threshold: float = 0.15
) -> DriftReport:
    """
    Compare la distribution des décisions baseline vs période courante.
    Retourne un signal si la divergence dépasse le seuil.
    """
    def distribution(decisions):
        total = len(decisions)
        counts = Counter(decisions)
        return {k: v / total for k, v in counts.items()}

    baseline_dist = distribution(baseline_decisions)
    current_dist = distribution(current_decisions)

    # Divergence totale sur toutes les catégories
    keys = set(baseline_dist) | set(current_dist)
    divergence = sum(
        abs(current_dist.get(k, 0) - baseline_dist.get(k, 0))
        for k in keys
    )

    detected = divergence > threshold
    signal = f"Divergence: {divergence:.3f} (seuil: {threshold})"

    return DriftReport(
        detected=detected,
        signal=signal,
        baseline_distribution=baseline_dist,
        current_distribution=current_dist,
        divergence=divergence
    )

# En production : check automatique quotidien
report = detect_decision_drift(
    baseline_decisions=load_baseline_decisions(),
    current_decisions=load_recent_decisions(days=7)
)

if report.detected:
    alert_team(f"Drift détecté — {report.signal}")
```

Ce code compare la distribution historique des décisions (approuvé, rejeté, escaladé) à la distribution des sept derniers jours. Si l'écart dépasse un seuil configurable, il déclenche une alerte. La logique est simple — volontairement. Ce qui compte ici, c'est la régularité d'exécution. Ce check doit tourner automatiquement, chaque jour, sans qu'on ait besoin d'y penser.

Un drift qui s'installe sur trois semaines avant d'être détecté, c'est trois semaines de décisions potentiellement incorrectes en production.

### Ce qu'on fait quand on détecte un drift

Détecter un drift n'est pas la fin du problème. C'est le début d'un diagnostic.

La première étape : identifier le mécanisme. Est-ce un changement de modèle ? Un changement de données entrant ? Un changement de comportement utilisateur ? La réponse détermine le traitement.

Un changement de modèle appelle un re-test systématique sur le jeu d'évaluation de 6.2. Si la nouvelle version du LLM dégrade les performances sur les cas de référence, on peut rester sur l'ancienne version (si le provider le permet), ou adapter les prompts pour compenser le changement de comportement.

Un changement de données appelle une révision des exemples d'évaluation. Les cas de référence doivent refléter la réalité actuelle du système, pas celle d'il y a six mois. Ajouter de nouveaux scénarios représentatifs. Relabeler si nécessaire. La baseline vieillit — il faut la rafraîchir périodiquement.

Un changement d'usage est le plus difficile à corriger parce qu'il révèle souvent une limite dans la conception initiale des règles. Il appelle une revue des prompts, parfois une refonte partielle de la logique de décision, et toujours une mise à jour des scénarios d'évaluation pour couvrir les nouveaux patterns.

Le principe commun à ces trois cas : ne jamais ajuster à chaud, sans re-tester sur le jeu de référence. Un correctif fait en urgence pour colmater un drift peut en créer un autre — sur des cas qui fonctionnaient correctement avant. Le remède peut devenir la cause suivante.

### Drift et rétrogradation automatique

Le mécanisme de rétrogradation posé en Partie 2 n'est pas réservé aux premières semaines de déploiement. Il s'applique tout autant à un agent en production stable depuis des mois.

Si un agent en phase Autonome détecte un drift significatif sur ses métriques décisionnelles, il doit rétrograder automatiquement en phase Supervisée. Pas attendre qu'un humain le remarque. Pas espérer que le phénomène se corrige spontanément.

Le drift non détecté transforme progressivement un agent fiable en agent incertain. Le drift non corrigé transforme un agent incertain en agent problème. Ces deux étapes prennent en général plusieurs semaines — ce qui est exactement le délai qu'une équipe met à remarquer que quelque chose ne va plus.

Le monitoring décisionnel de 6.3 est le filet. La détection de drift est le mécanisme d'alerte. La correction est le processus de remise en conformité. Ces trois couches ensemble forment un système qui se surveille activement.

Un agent n'est pas figé à son déploiement. Il évolue. Ce qui ne doit pas évoluer sans contrôle, c'est le comportement qui le définit — et les décisions qu'il prend au nom du système qui l'a déployé.
