## 4.5 Identity, historique, continuité

On a posé les fondations en 4.4. Trois stores, une couche mémoire unifiée, des interfaces `remember()` et `recall()` qui cachent la complexité. L'architecture tient.

Mais elle est incomplète.

Une architecture mémoire qui sait *comment* stocker sans savoir *à qui* appartient chaque souvenir, c'est une bibliothèque sans système de classement. Les livres sont là. Retrouver le bon au bon moment devient un pari.

Il manque trois choses. L'identité. La capacité à compresser l'histoire sans la perdre. Et le mécanisme concret pour reprendre là où on s'est arrêté.

### L'identité comme clé de voûte

Un agent sans notion d'identité traite tous les utilisateurs comme un seul utilisateur indistinct.

J'ai déployé un agent d'onboarding pour une startup SaaS d'une cinquantaine d'employés. Dans les premières semaines, il fonctionnait correctement — mais pour un seul utilisateur à la fois. Quand j'ai ouvert l'accès à toute l'équipe, les problèmes ont commencé.

L'agent rappelait à Marie les préférences de configuration que Thomas avait exprimées la veille. Il "se souvenait" de problèmes techniques qui n'étaient pas les siens. Dans quelques cas, il injectait des données métier appartenant à d'autres utilisateurs dans le contexte de quelqu'un d'autre.

Le diagnostic était évident en rétrospective : le memory layer ne distinguait pas les utilisateurs. Tout allait dans le même vector store, sans clé de partition. L'identité n'était pas un paramètre de la mémoire — c'était un oubli d'architecture.

La solution tient en deux identifiants. Le `user_id` lie les sessions au fil du temps. Le `session_id` lie les événements au sein d'une même session. Ces deux clés forment le squelette de toute architecture mémoire viable. Sans elles, `remember()` et `recall()` sont des fonctions sans adresse.

```python
# ❌ Mémoire sans identité — tout se mélange
def remember(content: str, metadata: dict) -> None:
    embedding = embedder.encode(content)
    vector_store.upsert(embedding, metadata)  # Aucune partition utilisateur

def recall(query: str, top_k: int = 5) -> list:
    return vector_store.search(query, top_k=top_k)  # Remonte des infos de n'importe qui


# ✅ Mémoire partitionnée par identité
def remember(content: str, user_id: str, session_id: str, metadata: dict) -> None:
    embedding = embedder.encode(content)
    enriched_metadata = {**metadata, "user_id": user_id, "session_id": session_id}
    vector_store.upsert(embedding, enriched_metadata)

def recall(query: str, user_id: str, top_k: int = 5) -> list:
    filters = {"user_id": user_id}  # Seule la mémoire de cet utilisateur est cherchée
    return vector_store.search(query, top_k=top_k, filters=filters)
```

Le filtrage par `user_id` fait plus que protéger la confidentialité. Il rend le retrieval plus précis — on réduit l'espace de recherche à ce qui est réellement pertinent pour cet utilisateur, ce qui améliore mécaniquement la qualité des résultats.

### L'histoire qui s'accumule

Avec l'identité en place, un nouveau problème émerge.

L'histoire s'accumule.

Après cent sessions avec le même utilisateur, le vector store contient des dizaines de conversations. Des informations utiles — mais aussi du bruit. Des préférences dépassées. Des problèmes résolus depuis longtemps. Des détails qui n'ont plus aucune valeur contextuelle.

Stocker tout sans filtre, c'est reproduire le problème qu'on a vu en 4.2 à une plus grande échelle. La pertinence future probable de chaque souvenir diminue avec le temps. Ce qui était signal devient bruit.

**Exemple concret :**

J'ai suivi un agent de coaching commercial déployé pour une équipe de douze personnes. Après six mois de production, les premières dérives ont commencé. L'agent conseillait des stratégies basées sur des informations de pipeline vieilles de trois mois. Il mentionnait des prospects que les commerciaux avaient depuis longtemps clôturés. Il rappelait des objectifs trimestriels qui n'étaient plus les leurs.

La mémoire long terme, jamais nettoyée, était devenue un passé figé qui contaminait le présent.

La réponse n'est pas de supprimer l'historique. C'est de le **compresser**. À la fin de chaque session, un processus de consolidation génère un résumé : ce qui a été accompli, les décisions importantes, les informations nouvelles apprises sur l'utilisateur. Ce résumé — compact, dense en signal — remplace le stockage brut des échanges dans le vector store long terme. La conversation disparaît. L'essentiel demeure.

```python
# ✅ Consolidation sélective avec résumé de session
def consolidate_session(session_id: str, user_id: str, conversation_history: list) -> None:
    if not should_consolidate(conversation_history):
        return  # Session trop courte ou sans contenu mémorisable

    # Générer un résumé structuré de la session
    summary = llm.generate(
        system_prompt=SUMMARIZATION_PROMPT,
        messages=[{
            "role": "user",
            "content": (
                "Résume cette session en retenant : décisions prises, "
                "faits appris sur l'utilisateur, sujets non résolus.\n\n"
                f"{format_conversation(conversation_history)}"
            )
        }]
    )

    # Stocker le résumé, pas la conversation brute
    remember(
        content=summary,
        user_id=user_id,
        session_id=session_id,
        metadata={
            "type": "session_summary",
            "date": datetime.now().isoformat(),
            "exchange_count": len(conversation_history)
        }
    )

    # Purger les embeddings bruts de cette session
    vector_store.delete(filters={"session_id": session_id, "type": "raw_exchange"})
```

La consolidation n'est pas une optimisation. C'est une décision architecturale sur ce que la mémoire long terme doit contenir. Des souvenirs édités, pas des transcriptions.

### Le démarrage à froid

L'identité est en place. L'historique est compressé. Reste la question concrète : comment démarre une nouvelle session ?

Sans protocole de démarrage, chaque session est une ardoise vide. L'architecture mémoire existe, mais elle n'est pas consultée au bon moment. L'agent repart de zéro par défaut — exactement le problème posé en 4.1, juste mieux dissimulé.

Le démarrage d'une session doit être un acte délibéré de reconstruction du contexte.

À l'initialisation, on charge d'abord le profil structuré de l'utilisateur depuis la base relationnelle — ses préférences, son statut, les données métier stables. Ensuite, on interroge le vector store pour retrouver les résumés de sessions récentes et les faits mémorisés pertinents. Ces éléments s'assemblent en un contexte de session qui donne à l'agent une base de départ réelle.

```python
# ✅ Initialisation de session avec reconstruction du contexte
def initialize_session(user_id: str) -> SessionContext:
    # 1. Profil structuré depuis la base relationnelle
    user_profile = db.get_user_profile(user_id)

    # 2. Résumés des sessions récentes
    recent_summaries = recall(
        query="historique récent et contexte actif",
        user_id=user_id,
        top_k=3,
        filters={"type": "session_summary", "recency": "last_30_days"}
    )

    # 3. Faits importants mémorisés sur cet utilisateur
    key_facts = recall(
        query="informations importantes et préférences de l'utilisateur",
        user_id=user_id,
        top_k=5,
        filters={"type": "user_fact"}
    )

    return SessionContext(
        user_id=user_id,
        session_id=generate_session_id(),
        profile=user_profile,
        prior_context=recent_summaries,
        key_facts=key_facts
    )
```

Cette initialisation transforme le "cold start" en démarrage informé. L'agent sait où il en est. Ou plutôt — il sait où l'utilisateur en est.

La différence perçue est immédiate. Plutôt que "Bonjour, comment puis-je vous aider ?", l'agent peut ouvrir avec "La dernière fois, on avait laissé en suspens votre migration vers la nouvelle version. Est-ce qu'on repart de là ?"

Ce n'est pas de la magie. C'est de la mémoire bien architecturée.

### L'architecture mémoire minimale viable

On peut maintenant assembler les pièces construites depuis 4.1.

[SCHÉMA 4.5] Architecture mémoire complète — identité, historique, continuité

L'amnésie fondamentale est irréductible. On ne supprime pas le stateless — c'est l'architecture des LLMs, pas un bug à corriger. Ce qu'on fait, c'est construire une couche mémoire qui contourne cette limitation par design. Chaque souvenir est attribué à une identité. L'historique est compressé au-delà d'un seuil. Chaque nouvelle session commence par une reconstruction explicite du contexte pertinent.

Ce n'est pas une mémoire parfaite. Un agent avec cette architecture ne devient pas omniscient. Il oublie encore des choses. Mais il répond à la question fondamentale posée en 4.1 : comment un système sans état natif crée-t-il de la continuité ?

Par construction délibérée.

Trois stores distincts, une couche d'abstraction, deux identifiants, un protocole de consolidation, un démarrage informé. Chacune de ces pièces est simple prise isolément. Assemblées, elles produisent quelque chose que l'utilisateur ressent comme de la continuité — et que l'agent peut maintenir sans effort à chaque session.

C'est le livrable de cette partie. Pas un agent parfait. Une architecture mémoire minimale viable qui tient en production.

---

*On a résolu le problème de la continuité. L'agent se souvient. Il sait à qui il parle. Il reprend le fil.*

*Mais la mémoire ne suffit pas pour produire un comportement fiable. Un agent qui se souvient de tout peut encore déraper dans son exécution — répéter les mêmes actions en boucle, accumuler des erreurs sans les détecter, s'engager dans une tâche sans plan cohérent. C'est le problème suivant. Et il vient d'un endroit inattendu : la boucle elle-même.*
