## Budget de contexte et priorisation

Voilà la situation. Le retrieval est corrigé. Le pipeline remonte maintenant les bons documents, dans le bon ordre, sans noyer le modèle. Mais il reste un problème.

Tout ne rentre pas.

Une fenêtre de 128K tokens, ça paraît immense. En production, le budget s'épuise vite. Le prompt système : 500 tokens. L'historique de conversation de la session en cours : 3 000 tokens. Les trois documents récupérés par le retrieval après reranking : 2 000 tokens. La requête utilisateur : 200 tokens. Les résultats des outils exécutés dans la boucle : 1 500 tokens.

On est à 7 200 tokens avant d'avoir généré la moindre réponse.

Sur une session longue, l'historique seul peut dépasser 20 000 tokens. Multiplie par des centaines d'utilisateurs en parallèle, et le "128K tokens, c'est illimité" devient très rapidement "128K tokens, il faut choisir ce qu'on garde".

La question n'est pas "Est-ce que ça rentre ?". Elle est : "Si ça ne rentre pas tout, qu'est-ce qu'on coupe ?"

### Ce qui ne se coupe pas

La première étape d'une gestion saine du budget, c'est de définir les priorités absolues. Ce qui doit toujours être là, quoi qu'il arrive.

Le prompt système est intouchable. C'est l'identité de l'agent, ses règles métier, ses contraintes de comportement. Si on le compresse ou qu'on en enlève des parties, le comportement dérive. On a vu en section 3.2 ce que ça donne en pratique : des agents bancaires qui oublient leurs règles de confidentialité, des agents support qui ne savent plus ce qu'ils peuvent promettre.

La requête utilisateur est intouchable. L'agent doit traiter ce que l'utilisateur demande — pas une version tronquée ou résumée de sa demande. Résumer la requête avant de la passer au modèle, c'est risquer de perdre le détail précis qui change tout.

Les documents retrieval de haute pertinence — les 3 à 4 retenus après reranking — sont prioritaires. C'est pour ça qu'on a construit un pipeline robuste en 3.4 : pour avoir peu de documents de haute qualité, pas beaucoup de documents bruyants. Ces documents gagnent leur place dans le contexte.

Tout le reste est négociable.

### Ce qui se compresse

Le premier levier, c'est la compression. Résumer les parties du contexte qui n'ont pas besoin d'être intégrales.

L'historique de conversation en est l'exemple canonique. Sur une session de 20 échanges, les 15 premiers n'ont généralement pas besoin d'être là mot pour mot. L'agent a besoin du fil conducteur — les décisions prises, les informations échangées, le périmètre établi — pas de la transcription complète.

**Exemple concret :**

J'ai travaillé sur un agent de support technique pour une plateforme SaaS. Les sessions pouvaient durer jusqu'à 30-40 échanges. Sans gestion du contexte, l'historique complet dépassait systématiquement 15 000 tokens. Résultat : l'agent perdait le fil des premières décisions, recommençait à poser des questions déjà posées, et produisait des réponses incohérentes avec ce qui avait été convenu en début de session.

La solution n'était pas de couper l'historique brutalement. C'était de le résumer progressivement. Au-delà de 8 échanges, les échanges anciens passaient dans un résumé structuré : "Problème identifié : erreur SSL sur le connecteur Salesforce. Actions déjà tentées : réinstallation du connecteur, vérification des certificats. Statut : non résolu." Deux cents tokens au lieu de 6 000. Toute l'information utile, sans le bruit.

```python
# ❌ Historique brut qui grossit sans limite
def build_context(history: list[dict], query: str, docs: list[str]) -> str:
    # L'historique complet : potentiellement des milliers de tokens
    history_text = "\n".join([f"{m['role']}: {m['content']}" for m in history])
    return f"{SYSTEM_PROMPT}\n\n{history_text}\n\nDocuments:\n{chr(10).join(docs)}\n\nUser: {query}"


# ✅ Historique compressé au-delà du seuil de rétention
def build_context(history: list[dict], query: str, docs: list[str]) -> str:
    RETENTION_WINDOW = 6  # Garder les 6 derniers échanges intacts

    if len(history) > RETENTION_WINDOW:
        old_exchanges = history[:-RETENTION_WINDOW]
        summary = summarize_exchanges(old_exchanges)  # Appel LLM léger
        recent = history[-RETENTION_WINDOW:]
        history_text = (
            f"[Résumé des échanges précédents]\n{summary}\n\n"
            f"[Échanges récents]\n"
            + "\n".join([f"{m['role']}: {m['content']}" for m in recent])
        )
    else:
        history_text = "\n".join([f"{m['role']}: {m['content']}" for m in history])

    return f"{SYSTEM_PROMPT}\n\n{history_text}\n\nDocuments:\n{chr(10).join(docs)}\n\nUser: {query}"
```

### Ce qui s'évince

Le deuxième levier, c'est l'éviction. Quand le budget est plein, quelque chose doit partir. Et ce quelque chose, c'est ce qui est le moins pertinent pour la décision à prendre maintenant.

La logique LRU (Least Recently Used) appliquée à l'historique — les échanges les plus anciens partent en premier après compression — est simple et fonctionne pour la plupart des cas.

Mais LRU seul n'est pas suffisant. Certains échanges anciens sont critiques : une contrainte exprimée en début de session ("je ne veux pas qu'on modifie le fichier de configuration"), une décision prise ensemble, une information de contexte unique. Évincer ces échanges par pur critère de date, c'est risquer de perdre des informations irremplaçables.

La distinction utile est entre deux types d'échanges : les échanges factuels, qui peuvent être résumés ou évincés sans perte critique, et les échanges décisionnels, qui doivent être préservés ou résumés avec une fidélité élevée. Cette séparation ne se fait pas automatiquement — elle se configure explicitement, souvent en taguant les échanges importants pendant la session.

### Ce qui se découpe intelligemment

Le troisième levier, c'est le chunking. Pas le chunking au moment de l'indexation — mais le découpage des documents au moment de leur injection dans le contexte.

Un document de 800 tokens récupéré par le retrieval n'a pas forcément besoin d'entrer en totalité. Si la requête porte sur un point précis d'une procédure, les 100 tokens pertinents sont plus utiles que les 800 tokens du document complet. Le reste est du bruit qui dilue l'attention du modèle — exactement ce qu'on a cherché à éviter depuis la section 3.1.

Ça suppose des chunks sémantiques — pas des découpes de taille fixe à 512 tokens, mais des unités cohérentes d'information : un paragraphe, une section, un article. Des blocs qui ont un sens complet isolément.

### Un framework pour allouer le budget

En pratique, voici comment je structure l'allocation du budget de contexte sur les agents que je déploie.

Le budget total est divisé en quatre zones. La zone réservée couvre le prompt système et la requête utilisateur — fixe, non compressible, toujours allouée en premier. La zone principale couvre les documents retrieval de haute pertinence — le cœur de l'information utile pour cette décision. La zone secondaire couvre l'historique récent de la session — configurable, compressible au-delà d'un seuil. La zone tampon couvre les résultats des outils et les métadonnées accumulées pendant la boucle.

```python
class ContextBudget:
    """Gestionnaire de budget de contexte avec priorités explicites."""

    def __init__(self, max_tokens: int = 8000):
        self.max_tokens = max_tokens
        # Allocation par zone (en % du budget total)
        self.zones = {
            "reserved":  int(max_tokens * 0.10),  # Prompt système + requête
            "primary":   int(max_tokens * 0.40),  # Documents retrieval
            "secondary": int(max_tokens * 0.35),  # Historique de session
            "buffer":    int(max_tokens * 0.15),  # Outils + métadonnées
        }

    def build(self, system_prompt: str, query: str,
              docs: list[str], history: list[dict]) -> str:
        # Zone réservée : toujours intégralement
        reserved = f"{system_prompt}\n\nUser: {query}"

        # Zone principale : top docs après reranking, jusqu'au budget
        primary = self._fit_docs(docs, self.zones["primary"])

        # Zone secondaire : historique compressé si nécessaire
        secondary = self._fit_history(history, self.zones["secondary"])

        return f"{reserved}\n\n{primary}\n\n{secondary}"

    def _fit_docs(self, docs: list[str], budget: int) -> str:
        result, tokens_used = [], 0
        for doc in docs:  # Déjà triés par pertinence décroissante
            doc_tokens = count_tokens(doc)
            if tokens_used + doc_tokens <= budget:
                result.append(doc)
                tokens_used += doc_tokens
            else:
                break  # On coupe, pas question de dépasser
        return "\n\n".join(result)

    def _fit_history(self, history: list[dict], budget: int) -> str:
        recent = history[-6:]  # Fenêtre de rétention
        recent_text = "\n".join([f"{m['role']}: {m['content']}" for m in recent])
        if count_tokens(recent_text) <= budget:
            return recent_text
        # Budget insuffisant même pour la fenêtre récente : compression forcée
        return summarize_exchanges(recent, max_tokens=budget)
```

### Le budget de contexte est une décision d'architecture

La gestion du budget n'est pas un problème d'optimisation technique. C'est une décision d'architecture qui reflète les priorités du système.

Décider que le prompt système est intouchable, c'est décider que le comportement de l'agent est non négociable. Décider que l'historique récent prime sur les documents retrieval pour certains cas d'usage, c'est une décision métier. Décider à quel seuil on compresse et ce qu'on résume, ça touche directement à la qualité des décisions.

Ces décisions méritent d'être explicites. Pas cachées dans un paramètre `max_tokens` quelque part dans le code. Documentées, versionnées, et révisées quand le comportement de l'agent dérive.

Le contexte n'est pas seulement une ressource technique à optimiser. C'est le reflet de ce que l'agent sait, de ce qu'il se souvient, de ce qu'on juge important pour ses décisions. Le gérer avec soin, c'est gérer la qualité du raisonnement de l'agent.

On a maintenant les outils pour traiter le contexte comme ce qu'il est : une ressource à allouer avec intention, pas un espace de stockage qu'on remplit jusqu'à débordement.

La prochaine erreur structurelle est encore plus fondamentale. On a parlé de ce que l'agent sait dans une session. Mais que se passe-t-il entre les sessions ? Un agent qui sait tout maintenant peut ne rien savoir dans dix minutes — quand la session se termine. L'agent est, par nature, amnésique.
