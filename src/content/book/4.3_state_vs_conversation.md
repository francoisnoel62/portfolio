# 4.3 State ≠ conversation

On a distingué court terme et long terme en 4.2.

Bien. Mais il reste une confusion plus insidieuse — et plus répandue. Une confusion que j'ai vue faire crasher des agents qui avaient pourtant une architecture mémoire correcte.

La confusion entre le **state** et la **conversation**.

Ce sont deux choses fondamentalement différentes. Mais dans la plupart des implémentations que j'ai croisées, elles sont traitées comme une seule et même chose. Et cette erreur-là est silencieuse. Elle ne casse pas immédiatement. Elle se révèle sous pression — quand la conversation s'allonge, quand l'utilisateur corrige quelque chose, quand un changement intervient en milieu de session.

### Ce que c'est, clairement

La **conversation**, c'est l'historique des échanges en langage naturel. Ce que l'utilisateur a dit. Ce que l'agent a répondu. Le fil du dialogue.

Le **state**, c'est l'ensemble des variables structurées qui décrivent la situation courante. Le statut de la tâche. Les données métier collectées. Les décisions prises. Les étapes franchies.

Ce ne sont pas des synonymes. Ce n'est pas parce qu'on peut *déduire* le state à partir de la conversation que le state *est* la conversation. Cette nuance est critique.

**Exemple concret :**

J'ai travaillé sur un agent de réservation hôtelière. Il gérait des conversations parfois longues — recherche de disponibilités, demandes spéciales, changements de dates, questions sur les options.

Au bout de quelques semaines en production, un pattern d'erreur a émergé. Quand un utilisateur modifiait ses dates en milieu de conversation — "en fait, je préférerais du 12 au 15" — l'agent continuait parfois sur les anciennes dates. Pas systématiquement. Aléatoirement, selon la longueur de la conversation et la formulation du changement.

Le diagnostic était simple : l'agent n'avait pas de state explicite. Pour connaître les dates de réservation, il relisait l'historique de conversation et tentait d'en extraire l'information à chaque appel. Sur une conversation courte, ça marchait. Sur une conversation de 20 messages avec des retours en arrière, le modèle perdait le fil.

Le state n'était pas stocké. Il était *inféré*. Et l'inférence sur un historique long est bruitée.

### Pourquoi l'inférence de state est une mauvaise idée

Lire la conversation pour en extraire le state, c'est demander au modèle de parser son propre historique de manière fiable à chaque requête.

Trois problèmes avec ça.

**Ambiguïté du langage naturel.** "Je préférerais le 15" — le 15 de quoi ? Du mois courant ? Du mois prochain ? L'agent doit inférer à partir du contexte. Parfois il se trompe. Et une erreur d'inférence sur le state, c'est une décision prise sur une fausse base.

**Fragilité sur les conversations longues.** Plus la conversation s'allonge, plus le signal se dilue. Le phénomène "lost in the middle" qu'on a vu en section 3.1 s'applique ici sans pitié : une information importante enterrée au milieu d'un historique de 30 messages n'est pas garantie d'être correctement interprétée.

**Coût à chaque requête.** Recalculer le state depuis l'historique, c'est une opération qu'on répète à chaque appel. C'est du bruit computationnel qui ne devrait pas exister — et qui introduit un point de défaillance à chaque itération.

### Le pattern : state explicite, conversation séparée

La solution est structurelle. Le state se maintient dans une structure de données dédiée — un objet, un dictionnaire, une entrée en base. Pas dans la conversation. Il est mis à jour de manière explicite quand quelque chose change. Il est injecté dans le contexte à chaque requête comme source de vérité.

La conversation reste ce qu'elle est : le fil des échanges, utile pour le ton, la continuité narrative, la compréhension des intentions. Mais elle ne porte pas le state.

```python
# ❌ State inféré depuis la conversation — fragile
def handle_message(conversation_history: list, message: str) -> str:
    # Le modèle doit déduire dates, statut et préférences depuis l'historique
    # En cas de conversation longue ou ambiguë : point de défaillance garanti
    response = llm.generate(
        messages=conversation_history + [{"role": "user", "content": message}]
    )
    return response


# ✅ State explicite + conversation séparée — robuste
@dataclass
class ReservationState:
    check_in: date | None = None
    check_out: date | None = None
    room_type: str | None = None
    guests: int = 1
    special_requests: list = field(default_factory=list)
    status: str = "pending"  # pending, confirmed, cancelled


def handle_message(
    state: ReservationState,
    conversation_history: list,
    message: str,
) -> tuple[ReservationState, str]:
    # Le state est injecté explicitement — source de vérité fiable
    state_context = f"""Réservation en cours :
- Check-in : {state.check_in}
- Check-out : {state.check_out}
- Type de chambre : {state.room_type}
- Voyageurs : {state.guests}
- Statut : {state.status}"""

    response = llm.generate(
        system=state_context,
        messages=conversation_history + [{"role": "user", "content": message}]
    )

    # Mise à jour explicite du state — pas d'inférence a posteriori
    updated_state = extract_and_apply_state_updates(response, state)

    return updated_state, response
```

Le state est une source de vérité. Il ne se déduit pas à la volée — il se maintient explicitement.

### Ce que ça change en pratique

Avec ce pattern, "je préférerais le 15" ne crée plus d'ambiguïté. L'agent met à jour `check_in = date(2024, 3, 15)` de manière explicite. La requête suivante injecte ce state mis à jour. L'historique de conversation peut être long, bruité, plein de corrections — le state reste propre.

C'est aussi ce qui rend la persistance inter-sessions possible. On l'a posé en 4.1 : la conversation disparaît à la fin de la session. Mais si le state est une structure externe, on peut le sauvegarder et le recharger. L'utilisateur revient le lendemain — on sait exactement où en est sa réservation. Pas besoin de reconstruire depuis un historique qui n'existe plus.

[SCHÉMA 4.3] State vs Conversation — deux couches distinctes dans l'architecture agent

Ce sont deux couches avec deux cycles de vie différents.

La conversation est éphémère par nature. Le state est structuré et persistant par design.

Les confondre, c'est construire sur une fondation qui tremble dès que la conversation devient complexe. Les séparer, c'est la décision architecturale qui rend l'agent fiable — en session, entre sessions, quelle que soit la longueur ou la complexité des échanges.
