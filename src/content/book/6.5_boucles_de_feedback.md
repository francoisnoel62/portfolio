## Boucles de feedback

On a posé chaque couche méthodiquement.

En 6.1 : l'évaluation après coup échoue. En 6.2 : l'approche eval-first, qui force à définir le succès avant de construire. En 6.3 : le monitoring décisionnel, qui suit le jugement, pas juste les réponses. En 6.4 : la détection du drift, qui compare les distributions pour repérer la dérive avant qu'elle fasse des dégâts.

C'est un système de surveillance solide.

Mais la surveillance seule ne suffit pas. Détecter un problème et le corriger sont deux choses différentes. Ce qui manque, c'est la boucle de retour : le mécanisme qui prend les signaux — drift détecté, décisions erronées, insatisfaction utilisateur — et les transforme en amélioration concrète.

C'est ça, une boucle de feedback. Et c'est ce qui fait la différence entre un agent statique qui se dégrade avec le temps et un système vivant qui apprend.

### Les trois sources de signal

Tout feedback n'est pas équivalent. J'en distingue trois types selon leur nature et leur fiabilité.

Le premier, c'est le feedback explicite. Ce que l'utilisateur exprime directement : un pouce vers le haut ou vers le bas, un commentaire, une correction manuelle. C'est le plus facile à collecter et le plus clair à interpréter. Un agent de support qui propose une procédure de remboursement et reçoit "ça ne correspond pas à ma situation" — c'est un signal direct, sans ambiguïté. Le problème du feedback explicite : il est rare. Les utilisateurs ne laissent pas de retour systématiquement. On obtient souvent de l'information sur les cas extrêmes — très bon, très mauvais — et presque rien sur les cas moyens, qui sont pourtant la majorité.

Le deuxième, c'est le feedback implicite. Ce que l'utilisateur fait, pas ce qu'il dit. Il a demandé à l'agent de rédiger un email — puis il l'a réécrit intégralement avant d'envoyer. Il a reçu une recommandation — puis il a choisi l'option opposée. Il a relancé immédiatement avec "non, ce que je voulais dire c'est...". Ces comportements sont des signaux. Ils sont moins précis que le feedback explicite, mais infiniment plus fréquents. Collectés à grande échelle, ils révèlent des patterns que le feedback explicite ne voit jamais.

Le troisième, c'est le feedback métier. Les KPIs avals. Est-ce que le contrat approuvé par l'agent a ensuite été contesté ? Est-ce que la note de frais validée était effectivement légitime ? Est-ce que le ticket trié comme "urgent" a été résolu dans les délais attendus ? Ce feedback est le plus tardif — il peut arriver des jours ou des semaines après la décision. Mais c'est le plus objectif. C'est lui qui dit si la décision était correcte en conditions réelles, pas sur un benchmark de laboratoire.

**Exemple concret :**
J'ai travaillé sur un agent de recommandation de contenu pour une plateforme de formation. On avait un taux de clics de 23% — pas mauvais. Mais en intégrant le feedback implicite (les utilisateurs qui cliquaient, commençaient le cours, puis abandonnaient dans les cinq premières minutes), on a découvert que l'agent recommandait des contenus qui suscitaient de la curiosité initiale mais pas d'engagement durable. Le vrai taux de pertinence était de 11%, pas 23%. Sans le feedback implicite, on aurait optimisé le mauvais indicateur pendant des mois.

### Le circuit de collecte

Collecter les trois types de feedback demande une architecture délibérée. Pas un formulaire ajouté après coup. Un circuit intégré dès la conception, branché sur le `DecisionLog` posé en 6.3.

```python
# ✅ Feedback lié à chaque décision loggée
from dataclasses import dataclass, field
from datetime import datetime
from typing import Optional, Literal

@dataclass
class FeedbackSignal:
    decision_id: str  # Lie le feedback à la décision dans DecisionLog
    signal_type: Literal["explicit", "implicit", "business"]
    value: float      # -1.0 (négatif) à 1.0 (positif)
    detail: Optional[str] = None
    captured_at: datetime = field(default_factory=datetime.now)

class FeedbackCollector:
    def __init__(self, storage):
        self.storage = storage

    def capture_explicit(self, decision_id: str, rating: float, comment: str = None):
        self.storage.save(FeedbackSignal(decision_id, "explicit", rating, comment))

    def capture_implicit(self, decision_id: str, user_action: str):
        negative = {"rewrote_output", "ignored_recommendation", "relaunched_immediately"}
        value = -0.5 if user_action in negative else 0.3
        self.storage.save(FeedbackSignal(decision_id, "implicit", value, user_action))

    def capture_business_outcome(self, decision_id: str, outcome_positive: bool):
        value = 1.0 if outcome_positive else -1.0
        self.storage.save(FeedbackSignal(decision_id, "business", value))
```

Ce n'est pas compliqué. C'est un changement d'habitude : chaque point de contact avec l'utilisateur et chaque outcome métier devient une opportunité de signal.

### De la collecte à l'amélioration

Collecter du feedback sans l'analyser, c'est remplir un entrepôt sans jamais ouvrir les cartons.

L'analyse a un objectif précis : identifier les patterns d'échec récurrents. Pas les erreurs isolées — les erreurs qui se répètent. Un agent de modération qui se trompe systématiquement sur les contenus sarcastiques. Un agent RH qui sous-estime régulièrement les cas avec plusieurs contrats successifs. Ces patterns ne sont visibles que dans les données agrégées.

Une fois le pattern identifié, trois leviers sont disponibles. Le premier, c'est l'ajustement de prompt — le plus rapide à tester, le moins stable dans le temps. Efficace pour corriger une formulation ambiguë, insuffisant pour traiter une lacune structurelle de raisonnement. Le deuxième, c'est la mise à jour des règles métier — ajouter une vérification explicite dans le pipeline pour les cas que le modèle gère mal. Moins fragile que le prompt, plus maintenable. Le troisième, c'est le fine-tuning — rarissime en pratique, mais pertinent quand le modèle doit intégrer un domaine très spécifique et que les deux premiers leviers ont montré leurs limites.

[DIAGRAMME 6.2] Boucle de feedback : collecte → analyse → amélioration → mesure → itération

La règle à retenir : chaque modification issue d'un feedback doit être validée sur le jeu de référence constitué en 6.2, avant toute mise en production. C'est la baseline qui empêche un correctif de créer un nouveau problème ailleurs. Et après déploiement, on surveille la distribution des décisions pendant 72 heures — le même protocole qu'en 6.4 pour la détection du drift.

### Le système d'évaluation continue

Voilà le livrable de la Partie 6.

Pas un outil isolé. Un système complet : évaluation dès le jour zéro, monitoring décisionnel en continu, détection du drift par comparaison de distributions, boucle de feedback qui ferme le cycle. Chaque couche alimente la suivante.

C'est ce qui transforme un agent statique en système qui apprend. Pas par magie. Par architecture.

Un agent n'est pas un prompt intelligent. C'est un système. Et un système qui dure en production, c'est un système qui sait ce qu'il décide, qui détecte quand il dérive, et qui intègre ce qu'il apprend pour s'améliorer. La boucle de feedback n'est pas la cerise sur le gâteau. C'est le mécanisme qui donne au système une durée de vie réelle.
