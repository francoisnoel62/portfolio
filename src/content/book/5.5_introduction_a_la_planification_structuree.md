## Introduction à la planification structurée

On a construit les couches une par une.

En 5.1, on a diagnostiqué ReAct naïf : une boucle aveugle qui tourne sans savoir si elle avance. En 5.2, les trois modes de défaillance autour des outils — boucles infinies, actions contradictoires, dépendances d'ordre. En 5.3, la propagation silencieuse des erreurs dans une chaîne de raisonnement, et le step_validator pour l'interrompre. En 5.4, l'idée centrale : un agent qui planifie avant d'agir est structurellement différent d'un agent réactif.

Il reste à formaliser. À passer des principes à un pattern codifiable, avec des phases explicites, des transitions définies, et des critères d'arrêt clairs.

C'est l'objet de cette section.

### Le pattern Plan → Execute → Verify

Le pattern tient en trois mots. Mais chaque mot cache une responsabilité architecturale distincte.

**Plan.** Avant d'appeler le moindre outil, le modèle génère une représentation structurée de ce qu'il va faire. Pas un raisonnement implicite caché dans sa chaîne de pensée — un artefact explicite, lisible, validable par le système et, si nécessaire, par un humain. Ce plan contient les étapes, leurs dépendances, et les conditions requises pour que chacune soit exécutable.

**Execute.** Le plan est déroulé séquentiellement. À chaque étape, le résultat est validé par le step_validator avant d'être injecté comme input de l'étape suivante. Si une étape produit une information qui invalide les étapes à venir, le système déclenche un replanning partiel — pas un abandon, pas une continuation aveugle.

**Verify.** Une fois toutes les étapes exécutées, une vérification globale répond à la question que la phase Execute ne peut pas poser : est-ce que l'objectif initial est atteint ? Si oui, on termine. Si non, on replanifie en tenant compte du contexte de l'échec.

C'est un cycle complet. Avec des critères d'entrée et de sortie explicites à chaque phase. Ce n'est pas une boucle infinie à qui on fait confiance pour s'arrêter.

### Phase 1 : un plan qui expose les bloquants

La valeur d'un plan ne se mesure pas à sa longueur. Elle se mesure à ce qu'il révèle avant toute exécution.

En demandant au modèle de formaliser ses étapes, leurs dépendances et les prérequis de chacune, on lui donne l'occasion d'identifier les bloquants — les conditions manquantes qui rendraient la tâche impossible à finaliser. C'est ce qui s'est passé avec l'agent de déplacements de la section 5.4 : un plan structuré aurait révélé l'absence d'outil pour la validation manager avant que les réservations soient confirmées et payées.

Un plan structuré a aussi une deuxième vertu : il est exposable. On peut le soumettre à un gating sur l'intention — montrer à un humain ce que l'agent prévoit de faire, avant qu'il le fasse. Un agent qui expose son plan n'est plus une boîte noire. C'est un système dont les intentions sont lisibles.

La stratégie de planification varie selon la nature de la tâche. Une tâche prévisible avec des étapes connues à l'avance appelle une planification one-shot : plan complet au départ, exécution séquentielle. Une tâche avec de l'incertitude — où des informations émergent en cours d'exécution — appelle une planification itérative : plan de haut niveau, puis révision à chaque jalon. Une tâche multi-domaines avec des sous-systèmes indépendants appelle une planification hiérarchique. Ce choix se fait à la conception, pas à l'exécution.

### Phase 2 : exécuter en maintenant la cohérence

La phase Execute n'est pas un passage en force.

C'est une boucle d'exécution instrumentée. À chaque étape, trois choses se passent : l'action est exécutée, le résultat est validé par le step_validator, et le plan est mis à jour pour refléter l'état réel. Cette mise à jour est importante. Elle permet de détecter en cours d'exécution une divergence entre ce que le plan prévoyait et ce que le monde a produit.

Si la divergence est locale — un outil qui retourne un format inattendu, un résultat légèrement différent de ce qui était anticipé — on adapte. Si la divergence est systémique — une hypothèse fondamentale du plan est fausse — on déclenche un replanning partiel sur les étapes affectées. Pas sur tout le plan.

Ce mécanisme de replanning partiel est ce qui distingue un système résilient d'un système fragile. ReAct naïf ne replanifie pas — il continue. L'agent planifié ajuste en cours de route, sur la base d'informations réelles.

### Phase 3 : vérifier que l'objectif est atteint

La phase Verify est la première qu'on supprime sous pression de délai.

C'est toujours une mauvaise idée.

Sans vérification globale, un agent peut terminer toutes ses étapes avec succès et rater l'objectif. Chaque appel d'outil a retourné 200. Chaque step_validator a validé. Et pourtant, la tâche initiale n'est pas accomplie — parce qu'une hypothèse de départ était incorrecte, parce qu'une contrainte non modélisée dans le plan a été violée, parce que le critère de succès était plus subtil que prévu.

La phase Verify répond à une question simple, définie au moment du Plan : est-ce que ce résultat satisfait l'objectif initial ? Si oui, le cycle se termine. Si non, on replanifie en intégrant le contexte de l'échec — les étapes qui ont produit des résultats incorrects, les hypothèses qui se sont révélées fausses. Et on recommence, dans la limite du nombre de cycles de replanification acceptables.

Ce critère d'arrêt global est ce qui rend le pattern fini. Et un système fini est un système qu'on peut monitorer.

### Le code du pattern complet

```python
# ❌ ReAct naïf — agit sans planifier, termine sans vérifier
def react_naive(task, tools, llm):
    context = [{"role": "user", "content": task}]
    for _ in range(15):
        response = llm.generate(context)
        action = parse_action(response)
        if action.type == "finish":
            return action.output  # Aucune vérification globale — on fait confiance au modèle
        result = tools[action.name](**action.args)
        context.append({"role": "tool", "content": str(result)})
    raise MaxIterationsError()


# ✅ Plan → Execute → Verify — trois phases, responsabilités séparées
def plan_execute_verify(task, tools, llm, max_replan_cycles=2):
    replan_cycles = 0

    while replan_cycles <= max_replan_cycles:
        # Phase 1 : PLAN
        plan = generate_plan(task, tools, llm)

        if plan.has_blockers():
            # Bloquant détecté avant toute action irréversible — on s'arrête proprement
            return PlanResult(status="blocked", blockers=plan.blocker_summary())

        # Optionnel : gating sur l'intention avant exécution
        # if requires_human_approval(plan):
        #     if not human_approves(plan):
        #         return PlanResult(status="rejected")

        # Phase 2 : EXECUTE
        execution_result = execute_plan(plan, tools, llm)

        # Phase 3 : VERIFY
        if verify_success(task, execution_result, llm):
            return PlanResult(status="success", result=execution_result)

        # Vérification échouée — on replanifie avec le contexte de l'échec
        task = enrich_task_with_failure_context(task, execution_result)
        replan_cycles += 1

    return PlanResult(status="failed", result=execution_result)


def execute_plan(plan, tools, llm):
    results = []
    for step in plan.steps:
        result = tools[step.action](**step.args)

        if not step_validator(step, result):
            # Résultat incohérent — replanning partiel depuis cette étape
            plan.replan_from(step.id, context=result)

        results.append(StepResult(step=step, result=result))
    return results
```

Ce qui frappe dans cette implémentation, c'est la séparation des responsabilités. La fonction `generate_plan` ne sait pas comment exécuter. `execute_plan` ne sait pas si la tâche est accomplie. `verify_success` ne sait pas comment corriger. Chaque phase fait une chose — et une seule. C'est ce qui rend le système compréhensible, testable, et observable.

### Ce que ça change en production

**Exemple concret :**
J'ai redéployé l'agent de déplacements professionnels de la section 5.4 avec ce pattern.

Avant : plan implicite dans la boucle ReAct. L'agent réservait l'hôtel et le vol avant de découvrir l'absence d'outil pour la validation manager. Résultat : deux réservations payées, tâche bloquée, intervention manuelle pour annuler.

Après : phase Plan. Le modèle liste les étapes, identifie la contrainte "validation manager requise avant envoi externe", détecte l'absence d'outil correspondant. Il retourne un bloquant — avant d'avoir touché à la moindre API de réservation. L'opérateur voit le bloquant. Il ajoute l'outil manquant. Le plan est réexécuté correctement, dans le bon ordre.

Sur l'ensemble des tâches de l'agent, les itérations moyennes sont passées de 10 à 6. La phase Verify a détecté trois cas où toutes les étapes avaient techniquement réussi mais l'objectif n'était pas atteint — des cas qui seraient passés en silence dans une boucle ReAct naïve. Moins d'appels API. Moins de tokens. Moins d'états incohérents à corriger a posteriori.

Le pattern n'a pas rendu l'agent infaillible. Il l'a rendu prévisible.

C'est le livrable de la Partie 5 : non pas un agent plus puissant, mais un agent dont le comportement est structuré, observable, et corrigible. Un agent qui, quand quelque chose ne va pas, peut montrer exactement où dans son plan le problème est apparu.

C'est ce qui sépare un système de production d'un prototype sophistiqué.

Un agent bien architecturé sait ce qu'il va faire avant de le faire. Il sait pourquoi chaque étape est nécessaire. Et quand il échoue, il sait pourquoi — et peut en informer le système.

Mais savoir qu'un agent fonctionne correctement à l'instant T ne dit rien sur ce qui se passe à T+2 semaines. Est-ce qu'il dérive ? Est-ce qu'il régresse sur certaines classes de tâches ? Est-ce que ses erreurs sont les mêmes qu'au départ, ou de nouvelles catégories d'échecs émergent-elles ?

C'est la question de la Partie 6. L'évaluation n'est pas un add-on qu'on branche quand le système semble stable. C'est une couche architecturale à part entière — et la seule façon de savoir si ce qu'on a construit tient vraiment dans le temps.
