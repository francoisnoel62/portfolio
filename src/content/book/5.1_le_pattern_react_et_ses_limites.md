## Le pattern ReAct et ses limites

Jusqu'ici, nous avons construit des fondations solides.

On sait quand ne pas utiliser un agent. On contrôle l'autonomie progressivement. On gère le contexte comme une ressource limitée. On a une architecture mémoire qui persiste entre les sessions.

Il reste un problème. La boucle elle-même.

ReAct — Reasoning + Acting — est le pattern fondateur des agents modernes. L'idée est simple : le modèle raisonne sur la situation, choisit une action, observe le résultat, recommence. Une boucle. Élégante sur le papier.

En démo, ça marche très bien. En production, c'est une autre histoire.

### Ce que fait ReAct, concrètement

Je l'ai posé en introduction : ReAct alterne raisonnement et action en boucle. Voici ce que ça donne dans le code le plus minimal possible :

```python
# ❌ ReAct naïf — le pattern qu'on voit partout en production
def react_loop(task, tools, llm):
    context = [{"role": "user", "content": task}]

    while True:  # Boucle infinie — aucun garde-fou
        response = llm.generate(context)
        action = parse_action(response)

        if action.type == "finish":
            return action.output  # On fait confiance au modèle pour s'arrêter

        result = tools[action.name](**action.args)  # Aucune validation du résultat

        context.append({"role": "assistant", "content": response})
        context.append({"role": "tool", "content": str(result)})
        # Et on recommence — sans vérifier si on avance vraiment
```

C'est exactement ce qui tourne derrière la majorité des agents en production. Et c'est exactement là que les problèmes commencent.

### Pourquoi ça marche en démo

Les démos sont conçues pour réussir. Pas par malhonnêteté — par construction.

En démo, le cas est connu à l'avance. L'environnement est stable. Le prompt a été calibré pour ce scénario précis. Les outils retournent ce qu'on attend. Et si quelque chose ne marche pas, on relance jusqu'à obtenir la bonne sortie.

J'ai présenté des démos d'agents qui semblaient impeccables. Cinq requêtes dans l'environnement de test, zéro accroc. Déployé en production deux semaines plus tard, le même agent échouait sur un tiers des tâches réelles.

La démo ne ment pas. Elle simplifie. Elle élimine tout ce qui fâche.

### Pourquoi ça casse en production

En production, il se passe trois choses que la démo occulte.

Premièrement, les inputs sont imprévisibles. Les utilisateurs posent des questions ambiguës, mal formulées, avec des contextes partiels. Le modèle raisonne avec ce qu'il a — et si l'input est flou, le raisonnement l'est aussi.

Deuxièmement, les outils échouent. Une API renvoie un timeout. Une requête SQL remonte un résultat vide ou tronqué. Un service externe est temporairement indisponible. Dans une boucle ReAct naïve, ces cas ne sont pas gérés. Le modèle continue à raisonner sur un résultat d'outil incorrect — sans le savoir.

Troisièmement, les erreurs s'accumulent. C'est le problème le plus sous-estimé, et je reviens dessus en détail dans la section 5.3.

**Exemple concret :**
J'ai déployé un agent de data analysis sur un système de gestion commerciale. Mission : répondre à des questions sur les performances des équipes de vente en interrogeant la base de données.

En démo, la tâche typique se déroulait parfaitement. L'agent interrogeait la base, récupérait les données du trimestre, calculait les ratios, renvoyait le classement des commerciaux.

En production, un lundi matin, voici ce qui s'est passé : l'agent a lancé une première requête SQL mal construite — une erreur de paramètre de date. Résultat : les données d'un seul mois au lieu du trimestre complet. L'agent n'a pas détecté l'anomalie. Il a raisonné sur des données incomplètes comme si elles étaient correctes, construit sa deuxième requête sur cette base erronée, et produit un classement commercial complètement faux — présenté avec la même confiance qu'un résultat valide.

```python
# ❌ Accumulation silencieuse d'erreurs
step_1 = tools.sql_query(
    "SELECT sales_rep, conversion_rate FROM sales WHERE month = ?",
    params=["2024-01"]  # Erreur : un seul mois au lieu du trimestre
)
# step_1 contient des données partielles — l'agent ne le sait pas

step_2 = tools.rank_by_performance(step_1)  # Classement sur données tronquées

step_3 = tools.format_output(step_2)  # Résultat final incorrect — aucune alerte
```

La boucle a continué. L'agent a raisonné sur chaque étape comme si la précédente était juste. C'est l'aveugle qui guide l'aveugle.

### Le vrai problème : ReAct est réactif, pas réflexif

Ce n'est pas que ReAct est un mauvais pattern. C'est qu'il est insuffisant seul.

La boucle Reason → Act assume que chaque étape est valide avant de passer à la suivante. Elle ne valide pas. Elle ne détecte pas les anomalies intermédiaires. Elle n'a aucun critère pour reconnaître qu'elle s'écarte de l'objectif global.

Il y a aussi un deuxième problème : les boucles infinies.

Sans limite d'itérations ni détection de cycles, un agent peut boucler indéfiniment sur une tâche impossible. J'ai vu un agent de planification de réunions tourner en boucle pendant 47 itérations parce qu'aucun créneau ne satisfaisait ses contraintes — et qu'il n'avait aucun mécanisme pour lever la main et dire "je suis bloqué". Il continuait à essayer. Encore. Encore. Jusqu'à ce qu'on coupe manuellement.

ReAct est réactif par nature. L'agent répond à ce qu'il observe, étape par étape. Mais il n'a pas de vue d'ensemble. Pas de plan. Pas de conscience de son propre état global.

En démo, ce n'est pas un problème : les cas sont simples, les boucles sont courtes. En production, c'est une bombe à retardement.

**Le problème :** La boucle ReAct naïve est aveugle à l'accumulation d'erreurs et incapable de s'arrêter quand elle déraille.

**La conséquence :** Des outputs incorrects produits avec une confiance normale, et des boucles infinies sur des cas impossibles.

**La solution :** Introduire de la structure — validation intermédiaire, limite d'itérations, détection de cycles, et planification avant l'action.
