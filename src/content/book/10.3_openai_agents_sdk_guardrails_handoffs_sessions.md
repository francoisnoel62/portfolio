## OpenAI Agents SDK : guardrails, handoffs et sessions

Les patterns de la Partie 7 (validation des entrées, isolation des sorties) et de la Partie 2 (gating, HITL) peuvent être construits à la main. L'OpenAI Agents SDK les industrialise : il réduit les primitives d'un système multi-agent à cinq concepts — Agent, Tool, Handoff, Guardrail, Session — et les câble ensemble.

L'intérêt n'est pas d'apprendre le SDK. C'est de comprendre ce que ces concepts modélisent, pour les reproduire quel que soit le framework.

### Les primitives de base

Un **Agent** est un LLM configuré avec un nom, des instructions système, et une liste de tools et d'handoffs disponibles. Le **Runner** exécute l'agent dans une boucle : appel LLM → appel tool → appel LLM → … jusqu'à un résultat final ou un handoff.

```python
from agents import Agent, Runner, tool

@tool
def get_invoice(invoice_id: str) -> dict:
    return fetch_from_db(invoice_id)

billing_agent = Agent(
    name="billing",
    instructions="Tu traites les questions de facturation. Ne discute pas d'autre chose.",
    tools=[get_invoice]
)

result = await Runner.run(billing_agent, "Montre-moi la facture #1042")
```

Simple. Mais la puissance est dans ce qu'on ajoute autour.

### Guardrails : contrôle avant et après le modèle

Un guardrail est une fonction de validation qui s'exécute en parallèle de l'agent (ou en amont, selon la configuration). S'il détecte un problème, il déclenche un **tripwire** — qui stoppe l'exécution avant que le résultat arrive à l'utilisateur.

Trois types :

- **InputGuardrail** : valide l'entrée utilisateur avant que l'agent ne la traite.
- **OutputGuardrail** : valide la sortie de l'agent avant de la retourner.
- **ToolGuardrail** : valide les paramètres d'un tool avant son exécution, et/ou son résultat après.

```python
from agents import InputGuardrail, GuardrailFunctionOutput

async def check_no_pii(ctx, agent, input) -> GuardrailFunctionOutput:
    if contains_pii(input.text):
        return GuardrailFunctionOutput(
            tripwire_triggered=True,
            output="PII detected in input"
        )
    return GuardrailFunctionOutput(tripwire_triggered=False)

agent_with_guard = Agent(
    name="safe-billing",
    instructions="...",
    input_guardrails=[InputGuardrail(guardrail_function=check_no_pii)]
)
```

C'est exactement le pattern de validation typée de 7.2 — séparer l'intention de l'instruction, valider avant d'agir. La différence : ici c'est déclaratif, pas procédural. Le guardrail s'exécute en parallèle de l'agent pour ne pas ajouter de latence.

Un détail important : les **input guardrails s'exécutent uniquement sur le premier agent** de la chaîne. Les output guardrails uniquement sur l'agent qui produit la réponse finale. Les tool guardrails, eux, s'exécutent à chaque invocation de tool — c'est là qu'on valide les paramètres avant une action irréversible (lien 2.4, gating).

### Handoffs : délégation typée d'état

Un handoff est le mécanisme par lequel un agent délègue une tâche à un autre, en lui transmettant un contexte typé. C'est du multi-agent sans magie noire : l'orchestrateur ne garde pas la main, il transfère explicitement.

```python
from agents import handoff

support_agent = Agent(
    name="support",
    instructions="Tu gères le support général. Si c'est une question de facturation, délègue.",
    handoffs=[handoff(billing_agent)]
)
# Quand support_agent décide de déléguer, il passe le contexte pertinent
# à billing_agent — et c'est billing_agent qui produit la réponse finale
```

Le handoff reste **dans un seul run**. Il ne crée pas un run séparé, ne sauvegarde pas d'état intermédiaire. C'est du routage d'agent, pas de l'orchestration durable (pour ça, voir 9.3).

Ce qui compte en production : les guardrails s'appliquent au périmètre de chaque agent. Si billing_agent a ses propres input guardrails, ils ne s'appliquent pas quand il reçoit un handoff de support_agent (les input guardrails ne s'appliquent qu'au premier agent). Il faut penser la chaîne de validation, pas juste l'agent terminal.

### Sessions : continuité inter-run

Le problème de la Partie 4 (l'agent est amnésique) a une réponse partielle dans le SDK : les **Sessions**. Une session relie plusieurs runs successifs d'un même agent sous un identifiant commun, en persistant automatiquement l'historique de conversation.

```python
from agents.extensions.handoff_filters import remove_all_tools
from agents import SQLiteSession

# En dev : session SQLite locale
session = SQLiteSession("user-42")

result = await Runner.run(agent, "Reprends notre analyse de ce matin", session=session)
# Le run précédent est disponible dans l'historique — sans le reconstruire manuellement
```

En production, on remplace SQLiteSession par une implémentation persistante (SQLAlchemy, Redis, ou l'API Conversations d'OpenAI). Le principe reste : **l'état est external au run** — exactement le point de 4.4.

### Human-in-the-loop : l'interruption comme primitive

Le SDK n'a pas de primitive HITL native au sens d'un vrai arrêt du run en attente d'approbation humaine. Ce qu'il offre : les guardrails peuvent retourner un résultat qui signale l'escalade, et le code appelant décide quoi faire.

```python
# ✅ Pattern HITL via guardrail + escalade externe
async def check_high_value_action(ctx, agent, tool_input) -> GuardrailFunctionOutput:
    if tool_input.get("amount", 0) > 10_000:
        # Pas de tripwire : on laisse l'agent continuer
        # mais on enregistre pour révision humaine
        escalation_queue.push({"run_id": ctx.run_id, "action": tool_input})
        return GuardrailFunctionOutput(tripwire_triggered=False)
    return GuardrailFunctionOutput(tripwire_triggered=False)
```

Pour un vrai HITL — agent suspendu jusqu'à réponse humaine — il faut l'orchestration durable de 9.3 : sauvegarder l'état du run, l'interrompre, reprendre après approbation. Le SDK ne le fait pas seul.

**Exemple concret :**
Un agent de traitement de notes de frais : input guardrail valide que la demande ne contient pas de données personnelles non pertinentes ; tool guardrail sur la fonction de remboursement bloque si le montant dépasse le plafond défini par rôle (lien 9.4, RBAC) ; handoff vers un agent de comptabilité pour les catégories spéciales. La chaîne tient en 60 lignes de code. Sans SDK, c'était 400 lignes de plomberie.
