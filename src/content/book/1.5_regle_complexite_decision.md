## Une règle. Une seule.

On a identifié les signaux. On a calculé le coût. On a vu les patterns d'échec.

Maintenant j'ai besoin de quelque chose de simple. Une règle que je peux appliquer en dix secondes avant de ouvrir un éditeur. Une question qui coupe court à tous les débats d'équipe sur "est-ce qu'on devrait utiliser un agent pour ça ?".

La voilà :

> **Utilise un agent si et seulement si la complexité de décision dépasse la complexité d'exécution.**

C'est tout.

Si l'aspect le plus difficile de ta tâche est de *savoir quoi faire* — raisonner sur un contexte ambigu, inférer des informations manquantes, adapter la réponse à l'imprévu — alors un agent est justifié.

Si l'aspect le plus difficile est de *faire* — orchestrer des appels API, transformer des données, coordonner des systèmes — tu as besoin d'un workflow. Pas d'un agent.

---

## Pourquoi cette règle fonctionne

Elle tranche parce qu'elle pointe vers la bonne dimension.

On se trompe souvent parce qu'on confond la complexité *apparente* d'une tâche avec la complexité *de la décision* qu'elle requiert. Une tâche peut être techniquement complexe — beaucoup d'intégrations, beaucoup de cas, beaucoup de données — et pourtant ne requérir aucun jugement contextuel réel.

C'est exactement ce qu'on a vu en 1.1 avec l'agent de gestion des congés. La tâche avait une vraie complexité d'exécution : intégrations SIRH, gestion des cas d'exception, règles métier nombreuses. Mais la complexité de décision était quasi nulle : les règles étaient documentées, les cas couverts, les exceptions cataloguées. Aucun jugement à exercer. Aucune ambiguïté à résoudre.

Résultat : un agent avec 12% de taux d'erreur là où un workflow aurait produit 0%.

**La règle :**

```
Complexité de décision > Complexité d'exécution → Agent
Complexité de décision ≤ Complexité d'exécution → Workflow ou Automation
```

---

## Mesurer la complexité de décision

La règle est simple. L'appliquer demande un peu de pratique.

Voici les questions concrètes à poser pour estimer la complexité de décision d'une tâche :

**1. L'entrée est-elle ambiguë ?**
Est-ce que l'agent va recevoir des inputs qui varient de façon imprévisible — formulations différentes, informations manquantes, contexte variable d'un utilisateur à l'autre ? Si oui, il y a de la complexité de décision.

**2. Les règles sont-elles explicites ?**
Peut-on écrire les règles de décision dans un document et couvrir 95% des cas ? Si oui, la complexité de décision est faible. Si les règles sont floues, contradictoires, ou dépendent d'un "jugement" humain difficile à formaliser, la complexité de décision est élevée.

**3. L'adaptation contextuelle est-elle nécessaire ?**
La réponse correcte change-t-elle significativement selon qui pose la question, dans quel contexte, avec quel historique ? Un agent de support qui répond à un utilisateur premium différemment d'un utilisateur standard, en tenant compte de l'historique de ses tickets et de la complexité de son problème — c'est de l'adaptation contextuelle réelle.

**4. L'imprévu est-il fréquent ?**
Quelle proportion des cas réels tombe en dehors des scénarios prévus au design ? Si moins de 5%, un workflow avec quelques branches d'exception suffit. Si plus de 20%, la tâche requiert un raisonnement adaptatif.

**Exemple concret :**

J'ai évalué deux projets en parallèle pour un client e-commerce.

*Projet A* : Un agent pour gérer les retours clients. Le client envoie un email, l'agent détermine si le retour est éligible selon la politique retour, et génère une étiquette ou un refus.

*Projet B* : Un agent pour répondre aux questions produit des clients. Le client pose une question sur un produit, l'agent cherche dans le catalogue, comprend la question souvent imprécise, et fournit une réponse adaptée au contexte d'usage du client.

Pour le Projet A, j'ai posé les quatre questions. L'entrée est structurée (commande + motif de retour). Les règles sont explicites (politique retour en 12 points). L'adaptation contextuelle est limitée (cas standard ou cas exceptionnel escaladé à un humain). L'imprévu est rare (moins de 5% des retours sortent du cadre). Verdict : workflow.

Pour le Projet B, même exercice. L'entrée est libre et souvent vague ("est-ce que ce truc convient pour un enfant de 6 ans ?"). Les règles ne s'appliquent pas — il faut comprendre l'usage implicite et la fiche produit. L'adaptation contextuelle est essentielle — la réponse change selon l'âge de l'enfant, l'usage prévu, les contraintes de sécurité. L'imprévu est la norme (chaque question est unique). Verdict : agent.

On a construit un workflow pour les retours. Délai de développement : huit jours. Taux d'erreur : inférieur à 1%.

On a construit un agent pour les questions produit. Délai de développement : quatre semaines. Satisfaction client sur les réponses : +34% comparé à la FAQ statique précédente.

Le bon outil pour le bon problème.

## La grille de décision

Pour rendre cette règle applicable immédiatement, voici la grille synthétique de la Partie 1.

[TABLEAU 1.3] Grille de décision : automation, workflow ou agent ?

```python
def choose_architecture(task: dict) -> str:
    """
    Grille de décision simplifiée.
    task = {
        "input_ambiguity": 0-3,     # 0 = structuré, 3 = très ambigu
        "rules_explicit": True/False, # règles documentées ?
        "contextual_adaptation": 0-3, # 0 = aucune, 3 = forte
        "unexpected_rate": float,    # % de cas hors-scénario
        "error_cost": "low"/"high",  # impact d'une erreur
    }
    """
    decision_complexity = (
        task["input_ambiguity"] +
        (0 if task["rules_explicit"] else 3) +
        task["contextual_adaptation"] +
        (task["unexpected_rate"] * 10)
    )

    if decision_complexity < 3:
        return "automation"

    if decision_complexity < 6:
        if task["error_cost"] == "high":
            return "workflow_with_human_validation"
        return "workflow"

    # Haute complexité de décision
    if task["error_cost"] == "high":
        return "agent_with_heavy_gating"
    return "agent"
```

Ce n'est pas un algorithme de production. C'est une aide à la réflexion. L'utiliser comme checklist mentale avant chaque décision architecture est suffisant.
