## Absence de planification

Valider avant de continuer. C'est ce qu'on a posé en 5.3.

Le step_validator empêche une hypothèse fausse de contaminer toutes les étapes suivantes. C'est nécessaire. Mais c'est encore réactif.

On corrige les erreurs après qu'elles sont apparues. On ne les prévient pas. Et dans une boucle longue, prévenir coûte infiniment moins cher que réparer.

### L'agent qui ne voit pas plus loin que son prochain pas

Un agent ReAct décide action par action. Chaque itération est une réponse à ce qui vient de se passer. Il n'a jamais de vue d'ensemble. Il ne sait pas combien d'étapes il lui reste. Il ne peut pas détecter à l'avance qu'une contrainte va bloquer son plan à l'étape 7.

J'ai déployé un agent chargé d'organiser les déplacements professionnels pour une équipe. La tâche était claire : réserver un hôtel, un vol, et envoyer les confirmations aux participants pour un déplacement à Lyon.

L'agent commence. Il cherche les hôtels disponibles à Lyon pour les dates demandées. Il en trouve un. Il réserve : succès. Il cherche les vols depuis Paris. Il en trouve. Il réserve également. Il prépare ensuite l'email de confirmation aux participants.

C'est là que ça coince.

La politique interne imposait une validation manager avant tout envoi de communication externe. L'agent n'avait pas d'outil pour déclencher ce processus. Il était bloqué — avec une réservation d'hôtel et un billet d'avion déjà confirmés, payés, sans pouvoir finaliser la tâche.

**Le problème :** l'agent avait exécuté des actions irréversibles avant de découvrir un blocage qu'il aurait pu identifier dès la première étape.

Un agent avec un plan aurait listé les étapes avant d'agir. Il aurait immédiatement vu la contrainte "validation manager requise". Et il aurait soit déclenché cette validation en premier, soit signalé le manque d'outil — avant de toucher à quoi que ce soit.

C'est la différence entre réactif et planifié.

### Penser avant d'agir

La planification dans le contexte des agents, c'est direct : avant d'exécuter quoi que ce soit, générer un plan structuré qui décrit les étapes, leurs dépendances, et les conditions requises. Le modèle utilise son raisonnement là où il est le plus fort — comprendre la situation globale — avant que les outils entrent en jeu.

```python
# ❌ Agent réactif — décide action par action, sans vue d'ensemble
def react_agent(task, tools, llm):
    context = [{"role": "user", "content": task}]
    iterations = 0

    while iterations < 15:
        response = llm.generate(context)
        action = parse_action(response)

        if action.type == "finish":
            return action.output

        result = execute_with_validation(tools, action)
        context.append({"role": "tool", "content": str(result)})
        iterations += 1

    raise MaxIterationsError("Tâche non terminée")


# ✅ Agent planifié — commence par un plan, détecte les bloquants avant d'agir
def planning_agent(task, tools, llm):
    # Phase 1 : générer un plan structuré
    plan_prompt = f"""
    Tâche : {task}
    Outils disponibles : {list(tools.keys())}

    Génère un plan d'exécution en JSON :
    {{
      "steps": [
        {{
          "id": 1,
          "action": "nom_de_l_outil",
          "description": "ce qu'on fait et pourquoi",
          "depends_on": [],
          "preconditions": ["conditions nécessaires avant cette étape"]
        }}
      ],
      "blockers": ["conditions manquantes qui rendraient la tâche impossible"]
    }}
    """
    plan = llm.generate_structured(plan_prompt)

    # Phase 2 : vérifier les bloquants AVANT d'exécuter quoi que ce soit
    if plan.blockers:
        return f"Impossible de procéder sans : {plan.blockers}"

    # Phase 3 : exécuter le plan dans l'ordre des dépendances
    results = {}
    for step in plan.steps:
        if not dependencies_satisfied(step, results):
            continue
        results[step.id] = execute_step(step, tools)

    return results
```

La différence n'est pas cosmétique. Dans la version planifiée, l'agent voit la totalité de la tâche avant de commencer. Il détecte les contraintes. Il ordonne les étapes correctement. Il identifie les bloquants avant de dépenser un seul appel d'outil.

### Trois niveaux de planification

Tous les agents n'ont pas besoin du même niveau de planification.

La planification one-shot génère un plan complet au départ et l'exécute en séquence. C'est adapté aux tâches prévisibles avec des dépendances stables. Le plan ne change pas en cours d'exécution — les étapes sont connues à l'avance.

La planification itérative commence par un plan de haut niveau, exécute quelques étapes, puis révise le plan en fonction de ce qu'elle a découvert. Elle convient aux tâches longues où des informations cruciales émergent en cours d'exécution. L'agent ne prétend pas tout prévoir — il révise régulièrement sa trajectoire.

La planification hiérarchique décompose une tâche complexe en sous-plans indépendants. Chaque sous-plan peut être planifié et exécuté séparément. C'est l'approche des agents qui coordonnent plusieurs flux en parallèle — un agent maître qui délègue à des agents spécialisés, chacun avec son propre plan local.

Le choix dépend de la complexité de la tâche et du degré d'incertitude sur les étapes à venir. Pour cinq étapes prévisibles : one-shot. Pour quinze étapes avec bifurcations possibles : itératif. Pour une tâche qui couvre plusieurs domaines : hiérarchique.

### Ce que la planification change concrètement

La planification ne ralentit pas l'agent. Elle l'accélère.

Parce qu'elle élimine les faux départs. Parce qu'elle évite les actions irréversibles exécutées avant que les conditions soient réunies. Parce qu'elle détecte, avant même la première action, les impossibilités qui auraient fait dérailler la boucle après six étapes de travail.

Sur l'agent de déplacement, les résultats ont été nets. Les tâches qui finissaient en erreur après 8 à 12 itérations réussissent maintenant en 5 à 7. Les cas vraiment impossibles s'arrêtent proprement dès le démarrage, sans avoir engagé la moindre réservation.

Moins d'appels API. Moins de tokens consommés. Moins d'états intermédiaires incohérents à nettoyer.

Il y a aussi un bénéfice moins visible mais tout aussi important : la lisibilité. Un agent qui génère un plan en début de tâche expose son raisonnement. On peut inspecter ce plan. On peut le valider avant exécution si la tâche est critique. C'est du gating sur l'intention, pas seulement sur les actions individuelles.

Un agent réactif est une boîte noire en mouvement. Un agent planifié montre où il va.
