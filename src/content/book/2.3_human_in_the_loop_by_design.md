## Human-in-the-loop by design

La question qu'on me pose le plus souvent, après avoir expliqué le modèle en trois phases : "Mais si on garde un humain dans la boucle, c'est encore vraiment un agent ?"

Oui.

Et c'est précisément ce qui le rend utilisable en production.

Le **human-in-the-loop** (HITL) — la capacité d'un agent à suspendre son exécution pour obtenir une validation humaine avant de continuer — n'est pas une limitation. Ce n'est pas un signe que le modèle n'est pas assez bon. C'est un choix architectural. Et comme tous les bons choix architecturaux, il doit être fait avant d'écrire la première ligne de code, pas en urgence après le premier incident.

### Le problème du HITL en afterthought

La plupart des équipes que j'ai vues intègrent le human-in-the-loop de la même façon qu'elles intègrent les tests : après coup, quand quelque chose casse.

L'agent fonctionne bien en démo. On le déploie. Un cas limite survient. L'agent prend une mauvaise décision avec des conséquences réelles. L'équipe ajoute une validation humaine en urgence, au mauvais endroit, dans le mauvais format, et ça crée plus de friction que de valeur.

J'ai vu un agent financier qui, après un incident, s'est retrouvé avec une validation humaine sur chaque transaction — y compris les virements de 3€ vers des fournisseurs récurrents. L'équipe avait paniqué et ajouté du contrôle sans réfléchir à la granularité. Résultat : les opérationnels passaient leur journée à approuver des transactions triviales, l'agent ne servait plus à rien, et six mois plus tard on l'avait désactivé.

Le HITL subi est un frein. Le HITL conçu est une force.

### Où placer les checkpoints

Tous les points de validation ne se valent pas. Le premier travail, quand on design un agent, c'est d'identifier les décisions qui méritent un arrêt.

Il y a une règle simple que j'utilise : **un checkpoint se justifie quand le coût d'une erreur dépasse le coût de l'interruption.** Pas avant.

Ce critère se traduit en trois catégories de décisions à gater par défaut :

Les **décisions irréversibles**. Envoyer un email à un client, supprimer un enregistrement, déclencher un paiement, publier un contenu — une fois fait, c'est fait. L'agent peut avoir raison 99% du temps. Le 1% restant justifie un arrêt.

Les **décisions à impact élevé**. Seuil financier au-delà d'un montant défini. Engagements contractuels. Modifications de données sensibles (RH, médical, légal). Plus l'impact est asymétrique, plus le checkpoint est justifié.

Les **décisions hors périmètre**. Quand l'agent reçoit une demande qu'il n'a pas été conçu pour traiter, il ne doit pas improviser — il doit escalader. Ce cas est plus subtil mais souvent le plus dangereux : l'agent répond avec confiance sur un terrain qu'il ne maîtrise pas.

### Deux types de validation : rapide et détaillée

Une fois qu'on a identifié où placer les checkpoints, il faut décider quelle forme de validation on demande à l'humain.

Il n'y a pas une seule façon de valider. Deux formats suffisent à couvrir la quasi-totalité des cas.

La **validation rapide** : l'humain approuve ou rejette une action en un clic, sans avoir à comprendre tous les détails. Format binaire. Idéal pour les décisions dont le contexte est évident et où la rapidité compte. "L'agent veut envoyer cette relance au client Dupont — OK ou Annuler ?"

La **validation détaillée** : l'humain doit fournir une explication, un contexte, ou une correction avant que l'agent puisse continuer. Format ouvert. Pour les décisions complexes, les cas ambigus, ou quand on veut capturer le raisonnement humain pour améliorer l'agent sur la durée.

**Exemple concret :**
J'ai travaillé avec une équipe RH qui utilisait un agent pour analyser des candidatures et proposer des shortlists. En phase supervisée, l'agent soumettait sa sélection avec une justification pour chaque candidat. Les recruteurs validaient d'un clic pour les cas clairs — candidat manifestement qualifié ou éliminatoire. Pour les cas limites, ils devaient noter pourquoi ils modifiaient la recommandation. Ces annotations ont alimenté trois révisions du prompt système en deux mois. À la fin, le taux de validation sans modification était passé de 71% à 89%.

Le HITL n'était pas qu'un filet de sécurité. C'était une boucle d'amélioration continue.

### L'implémentation concrète

Le système HITL le plus simple est aussi le plus robuste. Voici sa structure de base :

```python
# ❌ Agent sans HITL — autonomie totale par défaut
def execute_action(agent, action):
    result = agent.run(action)
    apply_to_system(result)
    return result

# ✅ Agent avec HITL structuré
def execute_action(agent, action, context):
    result = agent.run(action)

    if requires_human_validation(result, context):
        validation_type = get_validation_type(result)

        if validation_type == "quick":
            approved = request_quick_approval(result)
        else:
            approved, feedback = request_detailed_approval(result)
            if feedback:
                log_feedback_for_improvement(feedback, result)

        if not approved:
            return handle_rejection(result, context)

    apply_to_system(result)
    return result

def requires_human_validation(result, context):
    return (
        result.is_irreversible
        or result.financial_impact > context.approval_threshold
        or result.action_type in context.always_validate_actions
        or result.confidence_score < context.min_confidence
    )
```

Quatre conditions. C'est tout. Et c'est suffisant pour couvrir 90% des cas en production.

Notez le `confidence_score` dans la dernière condition. C'est souvent oublié, et c'est pourtant l'un des signaux les plus utiles. Quand l'agent lui-même n'est pas sûr de sa réponse — et les bons modèles savent l'indiquer — c'est souvent le signe qu'un humain doit regarder.

### Ce que le HITL révèle

Un détail que personne ne mentionne : le human-in-the-loop est aussi un outil de diagnostic.

Les décisions fréquemment rejetées par les validateurs humains sont un signal direct que quelque chose ne va pas dans le raisonnement de l'agent. Les décisions systématiquement approuvées sans modification signalent un périmètre qui pourrait être passé en automatique. Les décisions qui génèrent beaucoup de feedback détaillé indiquent des zones où le prompt ou les règles métier manquent de clarté.

Un agent sans HITL est une boîte noire. Un agent avec HITL est un système observable.

```python
# Logger les patterns de validation pour le diagnostic
def log_validation_outcome(result, approved, feedback=None):
    entry = {
        "action_type": result.action_type,
        "approved": approved,
        "had_feedback": feedback is not None,
        "confidence": result.confidence_score,
        "timestamp": now()
    }
    validation_log.append(entry)

# Analyser les patterns en régression
def analyze_hitl_patterns(log, window_days=14):
    rejection_rate_by_action = compute_rejection_rate(log, window_days)
    high_feedback_actions = find_high_feedback_actions(log, window_days)
    low_confidence_rejections = find_confidence_correlated_rejections(log)

    return {
        "candidates_for_automation": [
            a for a, r in rejection_rate_by_action.items() if r < 0.05
        ],
        "candidates_for_prompt_update": high_feedback_actions,
        "confidence_threshold_review": low_confidence_rejections
    }
```

Ce logging n'est pas du overhead. C'est le mécanisme qui permet de justifier la progression vers le niveau suivant — et qu'on a vu dans la section précédente comme condition de passage de phase Assistée à phase Supervisée.

### Le HITL comme signal de maturité

Il y a une ironie dans tout ça.

Les équipes qui résistent le plus au human-in-the-loop sont souvent celles qui en ont le plus besoin. Elles veulent l'autonomie avant d'avoir les données qui la justifient. Elles voient le HITL comme un signe que l'agent "ne marche pas encore bien".

Les équipes qui ont des agents en production depuis plus d'un an voient les choses différemment. Pour elles, le HITL est une fonctionnalité qu'on désactive progressivement, point de contrôle par point de contrôle, au fur et à mesure que les métriques le permettent.

Ce n'est pas de la prudence excessive. C'est de l'ingénierie.
