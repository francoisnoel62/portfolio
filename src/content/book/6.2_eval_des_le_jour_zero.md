## Eval dès le jour zéro

La section précédente a répondu au "pourquoi". L'évaluation ajoutée après coup échoue parce que le refactoring tardif ne récupère pas les données perdues — et que trois mois de comportement indétectable représentent des dégâts réels et irrécupérables.

Maintenant il faut répondre au "comment". Et surtout au "quand".

La réponse au "quand" est simple : avant la première ligne de code métier.

### Le renversement de perspective

Dans la plupart des projets, le cycle ressemble à ça. On code. On teste à la main. On déploie. On ajoute des logs "pour voir". On découvre les problèmes quand quelqu'un se plaint.

L'approche eval-first inverse cet ordre.

La première question n'est pas "comment je code ça ?". C'est "comment je vais mesurer que ça marche ?"

Ça paraît philosophique. Ce n'est pas philosophique. C'est une décision architecturale qui change tout ce qui suit.

Quand on se demande d'abord comment on va évaluer, on se force à définir ce que "marche" veut dire. Pour un agent de traitement de notes de frais, ça veut dire quoi exactement ? Que l'approbation correcte dépasse 95% ? Que les cas de fraude détectés représentent un seuil défini ? Que les rejets injustifiés restent sous une tolérance acceptable ?

Ces questions, si on ne les pose pas avant de coder, on ne les posera jamais correctement. Parce qu'une fois le code là, on a tendance à évaluer ce qu'on peut évaluer — pas ce qu'on devrait évaluer.

### Trois niveaux d'évaluation dès le premier jour

Quand je parle d'"eval dès le jour zéro", je ne parle pas d'un test unitaire sur la fonction `parse_expense`. Je parle d'une stratégie à trois niveaux qui se construit en parallèle de l'agent.

Le premier niveau, c'est l'évaluation des composants. Chaque outil, chaque décision isolée. Est-ce que la fonction de détection de fraude retourne ce qu'elle doit ? Est-ce que l'extracteur de montant parse correctement les formats edge cases ? Ce sont des tests déterministes. Les plus simples à écrire, les plus rapides à faire passer.

Le deuxième niveau, c'est l'évaluation des scénarios complets. Une note légitime qui devrait être approuvée. Une note hors politique qui devrait être rejetée. Une note ambiguë qui devrait déclencher une revue humaine. Ces scénarios doivent être rédigés avant de coder — pas construits en reverse-engineering à partir de ce que le code fait déjà.

Le troisième niveau, c'est l'évaluation humaine. Un échantillon de décisions reviewé manuellement chaque semaine. Pas pour remplacer les tests automatisés — mais pour capturer ce qu'ils ne voient pas. Les cas où le modèle donne la bonne réponse pour les mauvaises raisons. Les cas où le raisonnement est plausible mais la décision discutable. Ces deux patterns ne sont visibles qu'à l'œil humain.

**Exemple concret :**
J'ai travaillé avec une équipe qui construisait un agent de triage des tickets de support. Avant d'écrire la moindre logique d'agent, on a passé une demi-journée à constituer un jeu de 40 tickets labelisés : 15 urgents, 15 normaux, 10 ambigus. C'est ce jeu de données — pas le code — qui a guidé les décisions d'architecture. Quand on a eu une première version fonctionnelle, on l'a passé sur ces 40 tickets. L'agent se trompait sur 7 des 10 cas ambigus. Sans ce jeu de référence constitué avant, on n'aurait jamais su. On aurait déployé en pensant que "ça marche bien".

### Code : des tests qui existent avant le comportement parfait

Voilà à quoi ressemble l'approche eval-first dans la pratique. Ce n'est pas un test parfait au départ. C'est un test écrit au jour zéro qui grandit avec l'agent.

```python
# ✅ Eval dès le jour zéro — scénarios définis avant que le code soit parfait
from dataclasses import dataclass
import pytest

@dataclass
class EvalCase:
    ticket: dict
    expected_priority: str
    expected_reason_contains: str  # Fragment attendu dans le raisonnement

EVAL_CASES = [
    EvalCase(
        ticket={"content": "Service down, 500 clients impacted", "tier": "enterprise"},
        expected_priority="urgent",
        expected_reason_contains="business impact"
    ),
    EvalCase(
        ticket={"content": "Password reset request", "tier": "free"},
        expected_priority="normal",
        expected_reason_contains="standard"
    ),
    EvalCase(
        ticket={"content": "Invoice amount incorrect", "tier": "premium"},
        expected_priority="urgent",  # Finance + premium = urgent
        expected_reason_contains="billing"
    ),
]

def test_triage_decisions(triage_agent):
    correct = 0
    for case in EVAL_CASES:
        decision = triage_agent.process(case.ticket)
        if decision.priority == case.expected_priority:
            correct += 1
        assert case.expected_reason_contains in decision.reasoning.lower(), \
            f"Reasoning manque '{case.expected_reason_contains}' — ticket: {case.ticket['content']}"

    accuracy = correct / len(EVAL_CASES)
    assert accuracy >= 0.80, f"Accuracy {accuracy:.0%} sous le seuil — revoir la logique de triage"
```

Ce test ne passe pas forcément au jour zéro. C'est attendu. Ce qui compte, c'est qu'il existe. Il documente le comportement voulu. Il mesure l'écart entre la réalité et l'objectif. Il force à définir "80% d'accuracy minimum" comme critère de déploiement — avant d'avoir vu les chiffres, pas après.

C'est la différence entre un standard qu'on définit et un standard qu'on rationalise.

### La baseline : le cadeau qu'on se fait dans six mois

Il y a un bénéfice de l'eval dès le jour zéro qu'on sous-estime systématiquement.

La baseline.

Une baseline, c'est l'état de référence du système au moment de son premier déploiement. L'accuracy sur le jeu d'évaluation. La distribution des décisions par catégorie. Le taux d'escalation vers un humain. Ces chiffres, mesurés et documentés au jour zéro, deviennent dans six mois la seule façon de répondre à la question : "est-ce que l'agent a changé de comportement ?"

Comme nous l'avons vu en Partie 2 avec le drift comportemental, un agent peut dériver imperceptiblement — après une mise à jour du modèle, après un glissement dans les données d'entrée. Sans baseline, ce changement reste invisible jusqu'à l'incident. Avec une baseline documentée, on peut dire : "Au déploiement, on était à 94% d'accuracy sur les cas urgents. Aujourd'hui on est à 87%. Quelque chose a changé — voyons quoi."

Ce n'est pas de la paperasse. C'est de la mémoire institutionnelle pour un système qui n'en a pas naturellement.

L'eval dès le jour zéro, c'est ça : une stratégie à trois niveaux, un mindset qui inverse l'ordre de construction, et une baseline qui transforme six mois de production en données exploitables plutôt qu'en boîte noire.

Mais l'évaluation n'est pas statique. Un agent qui passe ses tests peut quand même prendre de mauvaises décisions en production — sur des cas que les tests ne couvrent pas, sur des comportements qui émergent sous charge, sur un raisonnement qui dérive en contexte réel.

L'étape suivante, c'est le monitoring continu des décisions.
