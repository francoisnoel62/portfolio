## Le protocole A2A : interopérabilité agent↔agent

MCP règle le problème de l'intégration aux outils. Mais un pattern revient de plus en plus en production : l'orchestrateur délègue une sous-tâche à un autre agent — qui tourne sur une autre stack, chez un autre éditeur, parfois dans une autre entreprise.

Comment deux agents communiquent-ils sans que chacun connaisse l'implémentation de l'autre ?

C'est le problème qu'**A2A (Agent-to-Agent)** résout. Le protocole a été annoncé par Google en avril 2025, lancé avec plus de 50 partenaires technologiques (Atlassian, Salesforce, SAP, ServiceNow…), et donné à la Linux Foundation en juin 2025 pour une gouvernance neutre. Il tourne sur HTTP, Server-Sent Events (SSE) et JSON-RPC 2.0 — exactement le même socle que MCP.

### Découverte : l'Agent Card

Avant de déléguer une tâche à un agent, il faut savoir ce qu'il peut faire. A2A résout la découverte avec l'**Agent Card** : un fichier JSON que chaque agent A2A-compatible publie à l'URL standardisée `.well-known/agent.json`.

```json
// Exemple simplifié d'Agent Card
{
  "name": "billing-agent",
  "description": "Handles invoice generation and payment processing",
  "version": "1.2.0",
  "capabilities": ["invoice.create", "payment.process", "refund.initiate"],
  "inputFormats": ["application/json"],
  "authentication": {
    "schemes": ["Bearer", "ApiKey"]
  },
  "endpoint": "https://billing-agent.internal/a2a"
}
```

Un orchestrateur peut donc découvrir les agents disponibles à la volée — en lisant leurs Agent Cards — sans documentation manuelle ni contrat d'intégration personnalisé. C'est le même principe que les DNS pour les serveurs : un mécanisme de découverte standard, pas un annuaire central.

### Communication : Tasks, Messages, Artifacts

A2A définit trois objets de base pour structurer la communication.

- **Task** : l'unité de travail. Un orchestrateur envoie une Task à un agent ; l'agent la traite et retourne un résultat. La Task a un identifiant unique (lien idempotence, 9.2), un état de cycle de vie (submitted / working / completed / failed) et un contexte.
- **Message** : les échanges au sein d'une tâche. Un agent peut envoyer plusieurs messages pendant le traitement (statut intermédiaire, questions de clarification). Le streaming se fait via SSE.
- **Artifact** : le résultat produit. Peut être du texte, un fichier, un objet JSON structuré.

```python
# ✅ Délégation typée via A2A
task = a2a_client.submit_task(
    agent_url="https://billing-agent.internal/a2a",
    task=Task(
        id=generate_idempotency_key(),   # lien 9.2
        type="invoice.create",
        input={"customer_id": cid, "amount": amount, "currency": "EUR"}
    )
)
# Suivi SSE
for event in task.stream():
    if event.type == "artifact":
        invoice_pdf = event.artifact
```

### Le problème de confiance inter-agent

A2A standardise le transport. Mais un orchestrateur qui délègue à un agent tiers pose une question de sécurité que le protocole ne résout pas automatiquement : **comment savoir que l'agent distant fait ce qu'il prétend faire ?**

Deux risques spécifiques au contexte multi-agent :

1. **Usurpation d'agent** : un attaquant publie une fausse Agent Card à `.well-known/agent.json`. L'orchestrateur croit déléguer à l'agent de facturation ; il envoie en réalité des données à un agent malveillant.
2. **Escalade de privilèges** : un sous-agent reçoit une tâche et tente d'invoquer des capacités non autorisées — en se présentant comme l'orchestrateur auprès d'un troisième agent.

La réponse est la même qu'en 9.4 (RBAC inter-tenant) : le contexte d'autorisation est **authentifié**, pas inféré. Chaque agent A2A agit avec les permissions explicitement déléguées par l'orchestrateur — pas celles qu'il revendique.

```python
# ❌ Naïf : faire confiance au claim de l'agent appelant
def handle_task(task):
    if task.caller_claims_admin:
        grant_elevated_access()   # n'importe qui peut mettre ce flag

# ✅ Correct : valider le token d'authentification, pas le claim
def handle_task(task, auth_token):
    actor = verify_bearer_token(auth_token)  # OAuth 2 / API key
    if "admin" in actor.scopes:
        grant_elevated_access()
```

### Ce que A2A apporte concrètement

**Avant A2A**, une architecture multi-agent maison avait un couplage fort : l'orchestrateur connaissait l'API interne de chaque sous-agent, le format de ses paramètres, ses codes d'erreur. Remplacer un sous-agent cassait l'orchestrateur.

**Avec A2A**, le contrat est la Task — pas l'implémentation. Remplacer l'agent de facturation v1 par v2 (ou par un agent d'un autre éditeur) ne change rien à l'orchestrateur, tant que la nouvelle Card déclare la même capability. C'est le même principe que les interfaces en programmation orientée objet, appliqué aux agents.

**Exemple concret :**
Un agent d'orchestration RH délègue trois sous-tâches en parallèle : vérification des antécédents (agent externe spécialisé), génération du contrat (agent juridique interne), planification de l'onboarding (agent calendrier). Sans A2A, chaque intégration est custom. Avec A2A, l'orchestrateur lit trois Agent Cards, envoie trois Tasks avec un idempotency key, et suit les statuts via SSE — indépendamment du fait que les trois agents tournent sur des stacks différentes.

### A2A et MCP : complémentaires, pas concurrents

MCP et A2A couvrent deux niveaux différents.

- **MCP** : agent ↔ outil (une entité sans LLM propre — filesystem, API, base de données).
- **A2A** : agent ↔ agent (deux entités avec leur propre LLM, leur propre planification, leur propre état).

Un même système peut utiliser les deux : l'orchestrateur communique avec ses sous-agents via A2A ; chaque sous-agent accède à ses outils via MCP. Les protocoles sont orthogonaux.
