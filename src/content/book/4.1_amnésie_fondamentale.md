## L'amnésie fondamentale

Voici ce qui se passait avec un agent de support interne qu'on avait déployé pour une équipe commerciale.

À chaque début de session, l'agent demandait aux commerciaux leur territoire, leur segment client, leurs produits clés. Des informations qu'ils avaient déjà fournies la semaine précédente. Et la semaine d'avant. Et celle d'avant encore.

L'équipe avait l'impression de parler à quelqu'un qui oubliait tout pendant la nuit.

C'est exactement ce qui se passait.

### Ce que "stateless" signifie vraiment en production

Les LLMs sont **stateless** — sans état. On l'a posé en introduction, mais ses implications concrètes méritent qu'on s'y attarde, parce que c'est le point de départ de tout ce que cette partie va construire.

Chaque appel à un LLM est indépendant du précédent. Le modèle ne conserve aucune information d'une inférence à l'autre. Il ne sait pas qu'il a répondu à cet utilisateur hier. Il ne sait pas que cette conversation a eu lieu. Les paramètres du modèle sont figés après l'entraînement — rien de ce qu'il "vit" pendant une session ne les modifie.

Ce n'est pas un défaut à corriger. C'est l'architecture.

Ce qui donne l'illusion de mémoire dans une conversation, c'est le contexte. On accumule les échanges dans la fenêtre de contexte, et le modèle peut s'y référer. Il semble se souvenir de ce qu'on lui a dit cinq messages plus tôt — parce que ce message est encore là, dans le prompt. Mais ce n'est pas de la mémoire. C'est du contexte. Temporaire, borné, volatile.

En section 3.3, on a établi que la fenêtre de contexte n'est pas une mémoire. Cette section pousse le raisonnement plus loin : qu'est-ce que ça veut dire, concrètement, pour un agent en production ?

### Le gouffre entre les sessions

Dans une session, l'illusion tient. L'utilisateur parle, l'agent répond, la conversation progresse. Ça ressemble à de la continuité.

Puis la session se termine. Timeout, fermeture d'onglet, lendemain matin. Le contexte disparaît. La session suivante commence sur une ardoise vide.

**Exemple concret :**

J'ai déployé un agent de diagnostic technique pour une entreprise de 300 personnes. L'objectif : aider les employés à résoudre leurs problèmes informatiques récurrents sans systématiquement solliciter le support.

Lors de la première session, un utilisateur a passé vingt minutes à décrire sa configuration complète — OS, version des logiciels, problème de connectivité intermittente avec le VPN depuis la mise à jour de novembre. L'agent a posé des questions précises, établi un profil technique détaillé, et résolu le problème immédiat.

Le lendemain, même utilisateur, même problème récurrent. L'agent a recommencé depuis zéro. Mêmes questions. Même protocole de diagnostic. L'utilisateur a envoyé un message que j'ai gardé en tête : "C'est quoi l'intérêt si je dois tout réexpliquer à chaque fois ?"

L'agent n'avait pas de "hier". Il avait ce qu'on lui donnait dans le contexte de la session courante. Rien de plus. Et on ne lui avait donné aucune mémoire persistante.

```python
# ❌ Agent stateless — chaque session repart de zéro
def handle_request(user_id: str, message: str) -> str:
    response = llm.generate(
        system_prompt=AGENT_SYSTEM_PROMPT,
        messages=[{"role": "user", "content": message}]
        # Aucune info sur l'utilisateur, aucun historique inter-sessions
    )
    return response


# ✅ Agent avec couche mémoire minimale
def handle_request(user_id: str, message: str, session_id: str) -> str:
    # Récupérer le profil persistant de l'utilisateur
    user_profile = memory_store.get_user_profile(user_id)
    # Récupérer le contexte des sessions précédentes pertinent
    cross_session_context = memory_store.get_relevant_history(user_id, query=message)

    enriched_messages = [
        {"role": "system", "content": build_system_with_profile(user_profile)},
        *cross_session_context,  # Ce que l'agent "sait" déjà sur cet utilisateur
        {"role": "user", "content": message}
    ]

    response = llm.generate(enriched_messages)

    # Persister l'échange pour enrichir les sessions futures
    memory_store.save_exchange(user_id, session_id, message, response)
    return response
```

### Pourquoi le prompting ne résout rien

J'entends souvent la même réaction : "Il suffit d'ajouter dans le prompt que l'agent doit mémoriser les informations importantes."

Le problème, c'est que le modèle peut bien produire "À retenir : l'utilisateur s'appelle Théo, problème VPN récurrent, config Windows 11" dans son output. Mais si personne ne lit cet output, ne l'extrait, ne le stocke dans un système persistant, la session suivante ne verra jamais cette information.

Le prompting déplace le problème. Il ne le résout pas.

L'amnésie fondamentale n'est pas une question de formulation. C'est une question d'architecture. La mémoire d'un agent n'est pas quelque chose qui existe nativement dans un LLM — c'est quelque chose qu'on **construit**, qu'on **persiste** et qu'on **injecte** délibérément dans chaque session.

[SCHÉMA 4.1] LLM stateless vs agent avec couche mémoire externalisée

Les sections suivantes vont construire cette couche mémoire pièce par pièce. Mémoire court terme, mémoire long terme, séparation entre state et conversation, externalisation dans les bons stores. Chaque dimension a ses outils, ses contraintes, ses trade-offs.

Mais le principe qui gouverne tout : **un agent sans architecture mémoire explicite n'a pas de mémoire. Il a des illusions de continuité.**

Et en production, les illusions ne tiennent pas longtemps.
