## Idempotence, retries et files de jobs

L'échec partiel de la section précédente a une réponse en trois temps : rendre les actions rejouables sans dommage, réessayer correctement, et sortir l'exécution longue de la requête HTTP.

### L'idempotence : réessayer sans dupliquer

L'**idempotence** (rejouer une opération produit le même résultat, sans effet en double) est la propriété qui transforme un retry dangereux en retry sûr.

On l'a effleurée en 5.2 côté conception d'outil : créer une ressource qui existe déjà ne doit pas lever d'erreur fatale ni créer un doublon — ça doit retourner l'existant. Mais ça ne suffit pas toujours, parce que l'agent ne sait pas forcément que « c'est la même action ». D'où la **clé d'idempotence** : un identifiant unique attaché à l'intention, pas à l'appel.

```python
# ❌ Sans clé d'idempotence : un retry = un doublon
def refund(order_id, amount):
    return payment_api.create_refund(order_id, amount)

# ✅ Avec clé d'idempotence : le retry est neutralisé côté service
def refund(order_id, amount, idempotency_key):
    # Le service de paiement déduplique : même clé = même opération, exécutée une seule fois
    return payment_api.create_refund(
        order_id, amount,
        idempotency_key=idempotency_key,   # ex. hash stable de (order_id, intent_id)
    )
```

La clé est générée *avant* l'appel, dérivée de l'intention (par exemple un hash de l'identifiant de tâche + l'action). Si l'agent réessaie après un échec partiel, il renvoie la même clé. Le service voit « déjà traité » et retourne le résultat initial. Le doublon de remboursement de la section 9.1 disparaît — pas parce que l'agent est plus prudent, mais parce que l'opération est idempotente.

Un mot pour couper court à un fantasme : l'« exactly-once » (exactement une fois) n'existe pas sur un réseau faillible. Ce qu'on peut garantir, c'est « au moins une fois » (on réessaie jusqu'au succès) combiné à l'idempotence (les rejouages sont neutralisés). Le résultat *observable* devient « effectivement une fois ». C'est une nuance que les ingénieurs distribués connaissent bien, et qu'on oublie systématiquement sur les agents : on ne supprime pas les doublons en priant pour que le réseau tienne — on les rend inoffensifs par conception.

### Réessayer intelligemment

« Réessayer » n'est pas une stratégie. C'est une catégorie de stratégies, dont la plupart sont mauvaises.

Trois règles :

1. **Ne réessayer que les erreurs retryables.** Un timeout, un 503, un 429 : oui. Une erreur de validation (4xx), une règle métier violée : non — réessayer ne changera rien, ça ne fera qu'amplifier le problème.
2. **Backoff exponentiel avec jitter.** Pas de retries serrés qui martèlent un service déjà à genoux. On espace : 1s, 2s, 4s… avec un peu d'aléatoire (jitter) pour éviter que mille agents réessaient en chœur (le « thundering herd »).
3. **Une limite, et une sortie.** Après N tentatives, on arrête et on route vers une **dead-letter queue** — une file des tâches « empoisonnées » qu'un humain ou un process dédié examinera. Réessayer à l'infini, c'est transformer une panne en boucle infinie coûteuse.

```python
# ✅ Retry correct : erreurs retryables, backoff + jitter, plafond, dead-letter
def retry_with_backoff(action, max_attempts=5):
    for attempt in range(max_attempts):
        try:
            return call_tool(action, idempotency_key=action.key)
        except RetryableError:
            if attempt == max_attempts - 1:
                dead_letter_queue.put(action)     # on n'insiste pas indéfiniment
                raise
            sleep(min(2 ** attempt, 30) + random_jitter())
        except FatalError:
            raise                                  # inutile de réessayer une 4xx
```

### Un agent n'a pas sa place dans une requête HTTP

Voici une erreur d'architecture que je vois constamment : faire tourner un agent à l'intérieur d'une requête HTTP synchrone.

Un agent qui planifie, appelle cinq outils, réessaie, vérifie, peut prendre 30 secondes ou trois minutes. Le mettre dans une requête HTTP, c'est garantir des timeouts de gateway, des tâches perdues au moindre redéploiement, et aucune capacité de reprise.

La bonne architecture : **l'exécution d'un agent est un job, pas une requête.**

- On *enqueue* la tâche dans une file durable (Redis, SQS, RabbitMQ, une table de jobs en base…).
- Un *worker* la traite en arrière-plan, avec retries et reprise.
- Le client *poll* le statut ou reçoit un webhook quand c'est fini.

[SCHÉMA 9.2] Requête → file de jobs durable → worker (retries, reprise) → notification ; vs agent dans la requête HTTP (fragile)

**Exemple concret :**
Un agent de traitement de documents tournait dans l'endpoint HTTP. Tant que les documents étaient courts, ça passait. Le jour où un client en a soumis un gros, l'agent a dépassé le timeout de la passerelle. La requête a été coupée — mais l'agent, lui, avait déjà déclenché la moitié de ses actions. État incohérent, impossible à reprendre. On a déplacé l'exécution dans une file de jobs : la même tâche est devenue un job rejouable, avec un statut interrogeable et une reprise propre. Le « gros document » est passé d'incident à non-événement.

### Ce que ça donne ensemble

Idempotence + retries corrects + file de jobs : on a un agent qui peut échouer, réessayer, redémarrer — sans dupliquer ni se perdre. C'est le socle de fiabilité.

Mais il reste un cas que l'idempotence seule ne couvre pas : quand l'agent enchaîne *plusieurs* actions qui doivent tenir ou tomber ensemble. Un vol réservé, mais pas l'hôtel. C'est le problème des transactions — et il n'a pas de solution simple.
