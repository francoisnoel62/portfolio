## Pourquoi l'évaluation après coup échoue

"On ajoutera le monitoring quand le système sera stable."

C'est la promesse qu'on fait tous. Je l'ai faite aussi.

Le problème, c'est que le système ne devient jamais stable de la façon dont on se l'imagine. Parce que la stabilité d'un agent n'est pas un état qu'on atteint — c'est quelque chose qu'on construit en mesurant, en corrigeant, en itérant. Un agent qu'on ne mesure pas n'est pas stable. Il est silencieux.

Et entre "silencieux" et "qui fonctionne bien", il y a un gouffre.

Dès l'introduction, j'ai posé cette erreur comme la sixième des erreurs structurelles : *ajouter l'évaluation après coup*. C'est la plus pernicieuse parce qu'elle ne bloque pas le déploiement. L'agent tourne. Les utilisateurs l'utilisent. Tout semble fonctionner. Et c'est exactement ce qui rend le problème invisible.

### L'agent tourne. Personne ne sait s'il fonctionne.

Voici ce que j'ai vu trois mois après le déploiement d'un agent de gestion des notes de frais dans une entreprise de services.

L'agent avait été déployé en production en septembre. En décembre, il traitait environ 200 demandes par semaine. Les utilisateurs semblaient satisfaits — les tickets de support n'avaient pas explosé, personne ne s'était plaint ouvertement.

Puis un responsable financier a posé la question : "Est-ce que votre agent valide correctement les dépenses hors politique ?"

Silence.

Personne ne pouvait répondre. Pas parce que l'équipe était incompétente — elle était excellente. Mais parce que l'agent n'avait jamais été instrumenté pour répondre à cette question. Il loggait les inputs et les outputs. Pas les décisions. Pas le raisonnement. Pas les cas de rejet.

Ce qu'on avait construit, c'était un agent qui *tournait*. Pas un agent dont on pouvait *évaluer* le comportement.

**Exemple concret :**
En fouillant les logs bruts, on a fini par reconstituer un échantillon de 50 décisions à la main. Résultat : l'agent approuvait les notes hors politique quand la description était formulée de manière vague. Environ 30% des cas. Un taux suffisamment bas pour passer sous le radar, suffisamment élevé pour représenter un problème financier réel sur 12 semaines de production.

Le problème n'était pas détectable sans métriques décisionnelles. Et ces métriques n'avaient pas été conçues dès le départ.

### Le coût du refactoring tardif

Ajouter l'évaluation après coup n'est pas une question de "juste rajouter quelques logs". C'est un refactoring architectural.

Un agent conçu sans instrumentation ressemble à ça :

```python
# ❌ Agent sans instrumentation — boîte noire
def process_expense(expense: dict) -> str:
    context = build_context(expense)
    decision = llm.generate(context)
    return decision  # Qu'est-ce qui s'est passé ? Mystère.
```

Quand on essaie d'ajouter l'évaluation après coup, on se retrouve à démonter une mécanique qu'on ne comprend plus bien. La décision est enfouie dans une chaîne de caractères. Le raisonnement n'a jamais été extrait séparément. Les cas de rejet n'ont pas été distingués des cas d'approbation dans la structure de sortie.

```python
# ✅ Agent conçu pour être évalué — observable dès la conception
@dataclass
class ExpenseDecision:
    expense_id: str
    approved: bool
    reasoning: str
    policy_violations: list[str]
    confidence: float

def process_expense(expense: dict) -> ExpenseDecision:
    context = build_context(expense)
    raw = llm.generate(context)

    decision = ExpenseDecision(
        expense_id=expense["id"],
        approved=parse_approval(raw),
        reasoning=parse_reasoning(raw),
        policy_violations=detect_violations(raw, expense),
        confidence=parse_confidence(raw)
    )

    log_decision(decision)  # Structuré, queryable, analysable
    return decision
```

La différence n'est pas dans la complexité du code. Elle est dans l'intention. Le second agent a été conçu pour être observable. Il retourne une structure, pas une chaîne. Chaque dimension de la décision est explicite et mesurable.

Refactorer le premier pour arriver au second, après trois mois de production et 2 400 décisions déjà prises ? C'est possible. C'est coûteux. Et ça ne récupère pas les données historiques — celles qui auraient pu montrer le problème bien plus tôt.

### Ce qu'on ne peut pas récupérer

C'est là que l'évaluation après coup échoue vraiment.

On peut refactorer le code. On ne peut pas retrouver le raisonnement de l'agent sur les décisions passées. On ne peut pas savoir si le comportement a dérivé progressivement sur trois mois. On ne peut pas identifier le moment précis où l'agent a commencé à approuver systématiquement ce qu'il aurait dû rejeter.

Comme nous l'avons vu avec le *drift comportemental* en Partie 2, un agent peut changer de comportement imperceptiblement — après une mise à jour du modèle, après un glissement dans les données d'entrée, après une évolution des usages. Sans baseline et sans mesure continue, ce changement est indétectable jusqu'à ce qu'il provoque un incident. Et à ce stade, les dégâts sont déjà là.

L'évaluation n'est pas quelche chose qu'on ajoute une fois que le système tourne.
C'est quelque chose qu'on conçoit avant qu'il tourne.

Et ça commence au jour zéro.
