## Monitoring des décisions, pas seulement des réponses

6.1 a posé le problème. 6.2 a répondu avec l'approche eval-first. Mais il reste une question que j'aurais dû me poser plus tôt : qu'est-ce qu'on monitore exactement, une fois l'agent en production ?

La réponse évidente — et la mauvaise — c'est : les réponses.

Latence. Taux d'erreur HTTP. Volume de tokens consommés. Qualité perçue du texte généré. Ces métriques sont faciles à collecter. Elles sont utiles pour l'infrastructure. Elles ne disent rien sur le comportement réel de l'agent.

La bonne réponse : les décisions.

### Réponse ≠ décision

Il faut clarifier cette distinction parce qu'elle change tout ce qui suit.

Une réponse, c'est ce que l'agent génère. Du texte. Une action. Un JSON. C'est ce qu'on observe en surface. C'est directement mesurable : le format est-il correct ? La réponse est-elle arrivée en moins de deux secondes ? L'API a-t-elle retourné 200 ?

Une décision, c'est ce que l'agent a *choisi*. Approuver ou rejeter. Escalader ou résoudre. Envoyer ou retenir. La décision se cache souvent derrière la réponse. Et c'est précisément là le problème.

**Exemple concret :**
Un agent de modération de contenu reçoit 500 messages par jour. Il retourne toujours quelque chose — "approuvé", "rejeté", "en revue". Le taux d'erreur HTTP est à zéro. La latence moyenne est de 340ms. Le système semble parfait.

Personne n'avait vérifié la qualité des décisions. Après six semaines, une plainte utilisateur a tout déclenché. En remontant les logs : l'agent approuvait systématiquement les contenus haineux formulés de manière détournée — et rejetait des contenus sarcastiques parfaitement légitimes. Des centaines de faux positifs et de faux négatifs, invisibles parce qu'aucune métrique ne capturait la décision elle-même.

La réponse était correcte. La décision était mauvaise. Personne ne le savait.

### Trois dimensions non négociables

Quand on monitore les décisions, on monitore trois choses distinctes.

La première, c'est quelle décision a été prise — pas la réponse textuelle, mais la décision structurée. "Approuvé" n'est pas une décision. "Approuvé, catégorie contenu marketing, risque faible, confidence 0.87" — c'est une décision. Elle est queryable. Elle est analysable. Elle peut être comparée à une baseline.

La deuxième, c'est pourquoi cette décision a été prise — le raisonnement. Pas le texte brut que le LLM génère, mais un extrait structuré du reasoning. Qu'est-ce qui a pesé dans la balance ? Quelle règle métier a été appliquée ? Quel signal a déclenché l'escalade ? Sans ce "pourquoi", on peut voir l'effet mais pas la cause. On ne peut pas diagnostiquer les erreurs récurrentes.

La troisième, c'est ce qui s'est passé ensuite — le résultat de la décision. L'utilisateur a-t-il contesté ? Un humain a-t-il corrigé ? Le workflow en aval a-t-il réussi ? Ce feedback de résultat est la seule façon de savoir si la décision était correcte en conditions réelles, pas sur un benchmark de laboratoire.

Ces trois dimensions ensemble permettent de répondre à des questions qu'on ne pouvait pas poser avec les métriques classiques : est-ce que l'agent prend les bonnes décisions ? Sur quels types de cas se trompe-t-il ? Son comportement a-t-il changé depuis la semaine dernière ?

### Le logging décisionnel structuré

Implémenter ce monitoring n'est pas sorcier. C'est un changement d'habitude plus qu'un problème technique.

```python
# ❌ Logging naïf — capture la réponse, pas la décision
def moderate_content(content: str) -> str:
    result = llm.generate(moderation_prompt(content))
    logger.info(f"Moderated: {result}")
    return result

# ✅ Logging décisionnel — capture la décision, le raisonnement, le contexte
from dataclasses import dataclass, field
from typing import Optional
import time

@dataclass
class DecisionLog:
    decision_id: str
    timestamp: float
    input_summary: str           # Résumé de l'entrée (pas les données brutes)
    decision: str                # La décision structurée
    confidence: float            # Score de confiance du modèle
    reasoning_excerpt: str       # Fragment clé du raisonnement
    triggered_rules: list[str]   # Règles métier qui ont joué
    escalated: bool              # Escalade vers un humain ?
    outcome: Optional[str] = None  # Renseigné en asynchrone via feedback

def moderate_content(content: str) -> ModerationDecision:
    start = time.time()
    raw_result = llm.generate(moderation_prompt(content))
    decision = parse_moderation_decision(raw_result)  # Structure typée

    log = DecisionLog(
        decision_id=generate_id(),
        timestamp=start,
        input_summary=summarize_for_logging(content),  # Pas les données brutes
        decision=decision.verdict,
        confidence=decision.confidence,
        reasoning_excerpt=decision.key_reasoning,
        triggered_rules=decision.applied_rules,
        escalated=decision.needs_human_review,
    )

    decision_store.append(log)  # Persisté, queryable
    return decision
```

Ce code fait une chose que le pattern naïf ne fait pas : il sépare la décision de la réponse. Le `verdict` est structuré. Le `confidence` est un signal. Le `key_reasoning` permet de comprendre le "pourquoi" en dix secondes. L'`outcome` peut être renseigné a posteriori quand le feedback arrive.

Et surtout : ces données sont queryables. Un simple `SELECT * FROM decisions WHERE confidence < 0.6 AND escalated = false` révèle immédiatement les cas où l'agent a décidé seul avec peu de certitude — exactement là où le risque est concentré.

### Les métriques qui comptent

Une fois les décisions loguées correctement, les métriques viennent naturellement.

Le taux de décisions correctes se mesure sur les cas où le résultat est connu — feedback humain, correction, contestation. Pas une estimation. Un chiffre réel, construit au fil des semaines.

Le taux de faux positifs et de faux négatifs doit être suivi séparément, jamais agrégé. Un agent de modération qui approuve 10% de contenu haineux et en rejette 10% de légitime a un taux d'erreur global de 20% — mais ces deux erreurs n'ont pas le même impact métier. Les agréger les cache.

Le taux d'escalade indique le niveau de confort de l'agent. Un taux trop élevé signale qu'il n'est pas à l'aise. Un taux trop bas dans les premières semaines de déploiement devrait inquiéter plus qu'un taux élevé.

La distribution des décisions dans le temps est le signal de drift le plus précoce. Si l'agent approuve 70% des cas en semaine 1 et 85% en semaine 6 sans raison visible, c'est un signal. Pas une certitude — mais un signal. Exactement le type de signal qu'on ne détecterait jamais en ne monitorant que la latence.

```python
# Rapport hebdomadaire sur les décisions loguées
def weekly_decision_report(store: DecisionStore, week: str) -> dict:
    decisions = store.query(week=week)

    return {
        "total": len(decisions),
        "approval_rate": rate(decisions, lambda d: d.decision == "approved"),
        "escalation_rate": rate(decisions, lambda d: d.escalated),
        "avg_confidence": mean([d.confidence for d in decisions]),
        "low_confidence_not_escalated": [
            d for d in decisions
            if d.confidence < 0.6 and not d.escalated
        ],  # Ces cas méritent une revue manuelle
        "outcome_known_rate": rate(decisions, lambda d: d.outcome is not None)
    }
```

Ce rapport ne remplace pas la revue humaine. Il la guide. Il indique où regarder en priorité. Et il documente la baseline contre laquelle le drift sera détecté — celui dont nous parlerons dans la section suivante.

### Ce que ça change

Un dashboard de monitoring d'agent n'est pas un dashboard de monitoring d'API. La confusion est fréquente. Elle est coûteuse.

Un dashboard d'API monitore la santé technique — latence, erreurs, disponibilité. Ces métriques sont presque binaires : ça marche ou ça ne marche pas.

Un dashboard de monitoring décisionnel monitore le jugement. C'est fondamentalement différent. Il répond à des questions qualitatives : est-ce que l'agent prend les bonnes décisions ? Sur quels cas se trompe-t-il ? Son raisonnement a-t-il dévié depuis la dernière mise à jour du modèle ? C'est une fenêtre sur le comportement de l'agent — et elle ne peut exister que si les décisions ont été loguées correctement depuis le premier jour.

Un agent dont on monitore les réponses peut tourner six mois avec un problème invisible. C'est le cas de la section 6.1 : l'agent des notes de frais approuvait 30% des cas hors politique. Personne ne le savait.

Un agent dont on monitore les décisions révèle ses dérapages en quelques jours. Pas parce que le problème est plus récent — parce qu'on regarde au bon endroit.

Le monitoring décisionnel, c'est ça : regarder au bon endroit. Pas la latence. Pas le taux d'erreur HTTP. Le jugement de l'agent. C'est ça qui compte en production.
