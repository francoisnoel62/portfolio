## Tracing, observabilité et sandboxing des outils

On a posé les fondations du tracing en 9.5 : un `trace_id` unique par run, corrélé au DecisionLog. Ce que la Partie 9 n'a pas détaillé, c'est le *comment* concret — quel format, quels outils, et comment les SDKs s'y branchent. Et il reste une question distincte : que se passe-t-il quand l'agent exécute du code ? Où tourne-t-il, avec quels droits ?

Deux réponses en 2025–2026 : **OpenTelemetry** pour le tracing, et les **sandboxes d'exécution** pour l'isolation.

### OpenTelemetry : le standard de tracing pour agents

**OpenTelemetry (OTel)** est le standard CNCF pour le tracing distribué, les métriques et les logs. Il s'est imposé dans l'infrastructure cloud ; il devient le standard de facto pour les agents aussi.

L'unité de base est le **span** : un bloc de temps avec un nom, des attributs, et un lien vers le span parent. Un run d'agent devient une arborescence de spans :

```
agent_run (trace_id=abc123)
├── llm_call (model="claude-opus-4-8", tokens=1247)
├── tool_call (tool="search_db", input={...}, attempt=1)
│   └── retry (attempt=2, backoff=2s)
├── tool_call (tool="generate_report", input={...})
└── decision_log (decision="escalate", rationale="amount > threshold")
```

Chaque span est exporté vers un backend — Jaeger, Honeycomb, Langfuse, ou n'importe quel collecteur OTel. Le `trace_id` relie tous les spans d'un même run, ce qui permet la reconstitution de bout en bout décrite en 9.5.

L'**OpenAI Agents SDK** implémente OTel nativement : chaque LLM call, tool invocation, handoff et guardrail trip est automatiquement tracé. On peut ajouter des spans custom pour les événements métier :

```python
from agents import trace

with trace("invoice_validation", trace_id=run.trace_id) as span:
    span.set_attribute("invoice.id", invoice_id)
    span.set_attribute("invoice.amount", amount)
    result = validate_invoice(invoice_id)
    span.set_attribute("validation.result", result.status)
```

Le trace_id produit ici est le même que le `trace_id` du DecisionLog de 6.3. Un incident déclenche une alerte → on ouvre le run dans le backend de tracing → on voit chaque étape. C'est exactement l'histoire reconstituable de 9.5.

### Attributs sémantiques : ce qu'on trace sur les LLM calls

OTel définit des **attributs sémantiques standardisés** pour les appels LLM — un groupe de travail de la fondation OTel y travaille depuis 2024. L'idée : tous les traces d'agents devraient utiliser les mêmes noms d'attributs pour les mêmes concepts.

```python
# Attributs standardisés sur un span de LLM call
span.set_attribute("gen_ai.system", "anthropic")
span.set_attribute("gen_ai.request.model", "claude-opus-4-8")
span.set_attribute("gen_ai.usage.input_tokens", 1247)
span.set_attribute("gen_ai.usage.output_tokens", 389)
span.set_attribute("gen_ai.response.finish_reason", "end_turn")
```

Résultat pratique : les dashboards de coût et de latence de 1.4 deviennent des requêtes standard sur les attributs OTel — pas des pipelines de parsing custom. Le budget de contexte de 3.5 est visible en temps réel.

### Sandboxing : isoler le code que l'agent exécute

Un agent qui génère et exécute du code pose le problème inverse du tracing : pas "que s'est-il passé ?" mais "que pouvait-il faire ?". En 7.3, on a posé le principe du sandbox — conteneur éphémère, egress allowlist. Voici les outils concrets.

**E2B** est le standard de facto pour exécuter du code généré par un LLM dans un environnement isolé. Chaque exécution tourne dans un microVM éphémère (basé sur Firecracker) avec :

- Filesystem volatile : rien ne persiste après la session.
- Réseau contrôlé : pas d'accès réseau par défaut ; on déclare explicitement les URLs autorisées.
- Timeout configurable : l'exécution ne peut pas tourner indéfiniment.

```python
from e2b_code_interpreter import Sandbox

async with Sandbox() as sbx:
    # Le code généré par l'agent tourne ici — pas sur le serveur
    result = await sbx.run_code(agent_generated_code)
    # La sandbox est détruite à la sortie du context manager
    # Le serveur hôte n'a jamais exécuté le code généré
```

C'est la mise en pratique du principe de 7.3 : le serveur de l'agent ne touche jamais le code généré. Il délègue à une sandbox, récupère le résultat, et continue.

**Daytona** pousse le concept plus loin : des environnements de développement complets, éphémères et reproductibles — utile quand l'agent doit générer, tester, et valider du code sur un projet entier plutôt que sur un snippet.

Pour les cas plus simples (pas de code généré, juste des outils qui appellent des APIs), un **conteneur Docker éphémère par tâche** avec réseau restreint suffit :

```python
# ✅ Tool qui tourne dans un conteneur éphémère
def run_in_sandbox(tool_code: str, allowed_egress: list[str]) -> str:
    container = docker_client.containers.run(
        "python:3.12-slim",
        command=["python", "-c", tool_code],
        network_mode="none",   # pas de réseau par défaut
        remove=True,           # conteneur détruit après exécution
        mem_limit="256m",
        cpu_quota=50000        # 50% d'un CPU
    )
    return container.logs().decode()
```

### Ce que le tracing ne remplace pas

OTel trace ce qui se passe. Il ne trace pas ce qui *aurait pu* se passer si le sandboxing n'avait pas été là.

Les deux couches sont complémentaires :
- **Sandboxing** : limite le rayon d'impact d'une exécution (sécurité préventive).
- **Tracing** : permet de reconstruire ce qui s'est passé après coup (sécurité detective).

Un agent sans sandbox mais avec tracing excellent : on verra très bien le désastre se dérouler, en détail, après coup. Un agent avec sandbox mais sans tracing : on saura qu'il a survécu, mais pas comment.

**Exemple concret :**
Un agent d'analyse de données qui génère du Python pour traiter des CSV uploadés par l'utilisateur. Sans sandbox, un utilisateur peut uploader un CSV dont le traitement déclenche un `os.system("rm -rf /")`. Avec E2B, le code s'exécute dans une microVM qui sera détruite dans 60 secondes quoi qu'il arrive — le serveur hôte est intact. Le span OTel montre que l'exécution a eu lieu, avec quels inputs, et ce qu'elle a retourné.
