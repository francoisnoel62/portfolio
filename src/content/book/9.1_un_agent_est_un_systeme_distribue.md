## Un agent est un système distribué (pas un script)

En démo, un agent ressemble à un script. Tu appelles le modèle, il choisit un outil, tu exécutes l'outil, tu renvoies le résultat. Linéaire. Propre. Trois lignes au tableau blanc.

En production, ce « script » est en réalité un système distribué. Et personne ne l'a décidé : ça vient avec les outils.

Parce que chaque appel d'outil n'est pas un appel de fonction. C'est un appel réseau vers un autre système — une API, une base, un service tiers — qui peut être lent, indisponible, saturé, ou pire : qui peut *réussir l'action mais perdre la réponse*. Dès qu'un agent a des outils, il hérite de tous les problèmes des systèmes distribués. Sans les avoir cherchés.

### Ce que la démo cache

Le piège, c'est que la démo marche. Toujours. Parce qu'elle s'exécute sur le chemin heureux : réseau stable, API qui répond, pas de concurrence, pas de panne.

La production, elle, vit sur les chemins malheureux. En voici les principaux :

- **Le timeout.** L'outil ne répond pas en deux secondes. L'agent attend-il ? Abandonne-t-il ? Réessaie-t-il ?
- **L'échec partiel.** Le plus vicieux. L'outil a exécuté l'action — créé l'enregistrement, envoyé le virement — mais la réponse s'est perdue en chemin. Côté agent : « échec ». Côté système : « fait ».
- **Le doublon.** Conséquence directe : l'agent réessaie une action déjà effectuée. Deux enregistrements, deux virements, deux emails.
- **La limite de débit.** Le service tiers renvoie un 429. L'agent doit ralentir, pas marteler.

**Exemple concret :**
Sur une intégration de données vers un système métier, l'agent appelait une API pour créer un enregistrement client. Le réseau a coupé juste après la création, avant que la réponse ne revienne. L'agent a vu un échec. Il a réessayé. Deuxième client créé. En une nuit de migration, des centaines de doublons — invisibles jusqu'à ce qu'un humain ouvre la base et découvre le carnage. Le code était « correct » : il gérait les erreurs. Il ne gérait pas le fait qu'une erreur réseau ne veut pas dire que l'action n'a pas eu lieu.

### Le bon modèle mental

Voici le retournement de cette partie : **arrête de penser ton agent comme un script, pense-le comme un système distribué.**

Ça change les questions qu'on se pose. On ne demande plus « est-ce que le code est correct ? » mais :

- Que se passe-t-il si cet appel échoue à mi-chemin ?
- Si je réessaie, est-ce que je risque de dupliquer un effet ?
- Si le worker crashe au milieu d'une tâche, où reprend-on ?
- Si dix agents tournent en parallèle, se marchent-ils dessus ?

```python
# ❌ L'agent-script : le chemin heureux, et rien d'autre
def run_agent(task):
    while not done:
        action = llm.next_action(context)
        result = call_tool(action)        # et si ça timeout ? si ça réussit sans répondre ?
        context.append(result)
    return context

# ✅ L'agent-système : chaque appel est faillible et explicite
def run_agent(task):
    while not done and steps < MAX_STEPS:
        action = llm.next_action(context)
        try:
            result = call_tool(
                action,
                timeout_s=action.timeout,
                idempotency_key=action.key,   # voir 9.2
            )
        except RetryableError as e:
            result = retry_with_backoff(action)   # voir 9.2
        except FatalError as e:
            result = handle_and_record(e)         # on n'avance pas sur une erreur fatale
        context.append(result)
        persist_checkpoint(context)               # on peut reprendre après un crash (voir 9.3)
    return context
```

Le second code n'est pas plus « intelligent ». Il est plus *honnête* : il admet que chaque ligne peut échouer, et il prévoit quoi faire.

### Les fausses certitudes

Il y a des hypothèses qu'on fait sans s'en rendre compte, et qui sont fausses en production : le réseau est fiable, la latence est nulle, l'outil répond toujours, un appel = un effet. Ce sont les vieilles « erreurs des systèmes distribués », et un agent les ramène toutes à la surface — avec une couche supplémentaire de non-déterminisme, parce que c'est un LLM qui décide *quel* appel faire.

Si vous venez du backend, rien de tout ça n'est nouveau : c'est le pain quotidien des systèmes distribués. Ce qui est nouveau, c'est que la séquence d'appels n'est plus écrite par un développeur — elle est décidée à la volée par un LLM. On ne peut donc plus raisonner sur les retries et l'idempotence en lisant le code une fois pour toutes : l'agent peut tenter une combinaison d'appels qu'aucun test n'a couverte. D'où une exigence renforcée — la fiabilité ne peut pas vivre dans le chemin que prend l'agent ; elle doit vivre dans les outils et l'infrastructure, qui, eux, restent déterministes.
