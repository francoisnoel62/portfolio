## Accumulation d'erreurs

Les trois modes de défaillance des boucles tool-use, on les a couverts en 5.2. Max iterations pour stopper les boucles infinies. Déduplication par signature pour les actions contradictoires. Validation des dépendances d'ordre pour les appels non-commutatifs.

Ces garde-fous sont nécessaires. Ils ne sont pas suffisants.

Il existe une catégorie d'échec plus insidieuse. Une où les outils n'échouent pas. Où les API retournent 200. Où les données arrivent proprement formatées. Et où l'agent produit quand même des résultats faux.

C'est la propagation d'erreurs dans le raisonnement enchaîné.

### La chaîne qui se rompt

Dans une boucle ReAct, chaque étape alimente la suivante. L'observation de l'étape N devient l'input du raisonnement de l'étape N+1. Ce raisonnement produit une action, qui génère une nouvelle observation, qui alimente l'étape N+2.

C'est une chaîne. Et comme toute chaîne, elle se rompt à son maillon le plus faible.

Sur une boucle courte — deux, trois étapes — les erreurs n'ont pas le temps de s'amplifier. Sur une boucle de six, sept étapes — ce que les tâches réelles en production requièrent — une petite erreur à l'étape 2 devient une grosse erreur à l'étape 6. L'agent ne s'en rend pas compte. Il continue à raisonner avec confiance sur des fondations qui tremblent.

Le résultat final ? Propre. Bien formaté. Plausible. Faux.

[DIAGRAMME 5.3] Propagation d'erreur dans une chaîne ReAct : une erreur à l'étape 2 contamine toutes les étapes suivantes

### L'agent de reporting qui ne sait pas qu'il se trompe

J'ai déployé un agent chargé de générer des rapports hebdomadaires de chiffre d'affaires. Cinq requêtes SQL enchaînées : récupérer les transactions de "la semaine dernière", les agréger par catégorie de produit, calculer les marges, comparer avec la période précédente, générer le résumé narratif.

L'étape 1 avait un bug subtil. L'agent calculait "les 7 derniers jours calendaires" au lieu de "la semaine business du lundi au vendredi". Trois jours de weekend inclus, une journée du vendredi précédent exclue.

Résultat : toutes les étapes suivantes étaient fausses. Les agrégations, les marges, la comparaison, le résumé. Chaque requête calculait juste, sur les mauvaises données.

Aucun outil n'a échoué. Aucune exception levée. L'agent a livré un rapport complet, cohérent en apparence, avec des chiffres plausibles. Personne n'a vu l'erreur pendant deux semaines, jusqu'à ce qu'un contrôleur remarque un écart avec les chiffres de l'ERP.

Deux semaines de mauvaises décisions. Pour un mauvais découpage de dates à l'étape 1.

**Le problème :** chaque étape avait traité l'output de l'étape précédente comme une vérité. Ni le code, ni l'agent ne s'étaient demandé : "Est-ce que ces données ont du sens ?"

### Le pattern : chaque résultat est une hypothèse

Voilà ce qu'un ReAct naïf fait implicitement : il traite chaque observation comme un fait établi.

```python
# ❌ Pattern naïf : confiance aveugle dans les observations
def react_loop(task, tools, llm):
    context = [{"role": "user", "content": task}]

    while True:
        response = llm.generate(context)
        action = parse_action(response)

        if action.type == "finish":
            return action.output

        result = tools[action.name](**action.args)
        # Résultat injecté directement — aucune vérification de cohérence
        context.append({"role": "tool", "content": str(result)})
        # L'étape suivante raisonne sur ce résultat comme s'il était vrai


# ✅ Pattern robuste : validation intermédiaire avant de continuer
def react_loop(task, tools, llm, step_validator=None):
    context = [{"role": "user", "content": task}]

    while True:
        response = llm.generate(context)
        action = parse_action(response)

        if action.type == "finish":
            return action.output

        result = tools[action.name](**action.args)

        # Vérifier la cohérence du résultat avant de l'injecter
        if step_validator:
            check = step_validator.validate(action, result, context)
            if not check.is_valid:
                # Signaler l'anomalie — ne pas continuer aveuglément
                context.append({
                    "role": "system",
                    "content": f"Résultat suspect à l'étape '{action.name}' : {check.reason}. "
                               f"Vérifier avant de continuer."
                })
                continue  # L'agent va raisonner sur l'anomalie

        context.append({"role": "tool", "content": str(result)})
```

La différence n'est pas dans la gestion des erreurs techniques. C'est dans la validation de la **cohérence** des résultats.

### Ce qu'on valide, concrètement

Pas besoin d'un système complexe. Quelques assertions au bon endroit changent tout.

Pour une requête SQL : le nombre de lignes retournées est-il dans la fourchette attendue ? La plage de dates couvre-t-elle bien la période demandée ? Les agrégats ont-ils des ordres de grandeur plausibles ?

Pour un résultat d'API : les champs critiques sont-ils présents et non-nuls ? Le format correspond-il à ce qu'on attendait ?

Pour un calcul intermédiaire : le résultat est-il dans l'intervalle possible ? Une marge négative de 40% sur un produit standard mérite d'être questionnée avant de devenir la base du calcul suivant.

La règle est simple. Chaque résultat intermédiaire est une **hypothèse à vérifier**, pas un **fait à propager**. Un outil qui retourne 200 confirme que l'appel technique a réussi. Il ne dit rien sur la justesse du résultat.

C'est une distinction que la boucle ReAct naïve ne fait pas. C'est une distinction que les agents en production doivent apprendre à faire.

L'accumulation d'erreurs ne s'arrête pas d'elle-même. Elle s'arrête quand on décide de valider avant de continuer.
