## Tools, sandboxing et moindre privilège

En 7.2, on a posé le principe : réduire le rayon de souffle. Cette section répond à la question concrète — *où* ce rayon se définit-il ?

Réponse : dans les outils.

Un LLM seul ne fait rien. Il produit du texte. C'est l'outil branché derrière qui transforme ce texte en action : une requête SQL, un virement, un email, un appel système. La couche d'outils, c'est le bras armé de l'agent. C'est donc là que se joue l'essentiel de sa sécurité.

### L'excessive agency, en trois formes

L'OWASP (LLM06) décompose l'excès d'autonomie en trois excès distincts. Les nommer aide à les diagnostiquer.

– **Trop de fonctionnalités.** L'agent a accès à un outil qui fait plus que nécessaire. Un outil « exécuter du SQL arbitraire » alors qu'il faut juste lire une commande client.
– **Trop de permissions.** L'outil tourne avec des droits trop larges. Un agent de lecture connecté à la base avec un compte qui peut aussi écrire et supprimer.
– **Trop d'autonomie.** L'agent peut déclencher une action irréversible sans aucune validation.

Ces trois excès sont cumulatifs. Et c'est leur combinaison avec une injection (7.2) qui transforme une manipulation en incident.

### Levier 1 : scoper les capacités

La première règle est ennuyeuse et décisive : **un agent ne doit avoir accès qu'aux outils strictement nécessaires à sa tâche, et chaque outil doit faire le minimum.**

Pas d'outil « couteau suisse ». Un outil générique est un outil dangereux.

```python
# ❌ Excessive agency : un outil qui fait tout
def run_sql(query: str):
    return db.execute(query)   # l'agent peut lire, écrire, supprimer, tout

# ✅ Des outils étroits, à intention unique
def get_order_status(order_id: str) -> OrderStatus:
    # requête paramétrée, lecture seule, périmètre borné
    return db.read_only.fetch_order_status(order_id)
```

Le second outil ne *peut pas* faire de dégâts, quelle que soit l'instruction reçue. Il n'expose qu'une intention : lire un statut de commande. Aucun prompt, aussi malveillant soit-il, ne lui fera supprimer une table — la capacité n'existe pas.

### Levier 2 : le moindre privilège jusqu'à la base

Scoper l'outil ne suffit pas si l'outil tourne avec des droits de super-utilisateur. Le moindre privilège doit descendre jusqu'à l'infrastructure.

– L'outil de lecture utilise un compte base **en lecture seule**.
– L'agent qui traite les commandes du client A ne peut pas, techniquement, toucher aux données du client B (cloisonnement, qu'on retrouvera dans le multi-tenant en Partie 9).
– Les clés d'API données à un outil sont *scopées* : un token qui ne peut appeler qu'un endpoint précis, jamais une clé maîtresse.

**Exemple concret :**
Un agent d'assistance technique avait un outil de redémarrage de services. Pratique — sauf que le token utilisé avait les droits sur tout le cluster. Le jour où une injection indirecte (via un log piégé qu'il analysait) lui a demandé de « redémarrer tous les services pour appliquer la mise à jour », il en avait la capacité. On a réduit le token à un seul service, derrière une allowlist. Même instruction, ensuite : refus, parce que la permission n'existait plus. Le correctif n'était pas dans le modèle. Il était dans le périmètre du token.

### Levier 3 : sandboxer l'exécution

Certains agents exécutent du code, lancent des commandes, ouvrent des URL. C'est puissant, et c'est la porte ouverte si l'exécution se fait dans le même environnement que l'application.

Le **sandboxing** (exécution isolée, sans accès au système hôte ni au réseau interne) n'est pas un luxe ici. C'est la condition pour qu'un agent qui exécute du code ne soit pas un agent qui exécute *n'importe quel* code n'importe où.

Quelques principes concrets :
– Exécution dans un conteneur jetable, sans secrets montés, sans accès au réseau interne.
– Egress réseau par **allowlist** : l'agent ne peut joindre que les domaines explicitement autorisés (clé de l'anti-exfiltration, voir 7.4).
– Quotas de temps, de mémoire, d'appels — pour éviter la consommation incontrôlée (LLM10).
– Aucun montage de fichiers sensibles dans l'environnement d'exécution.

```python
# ✅ Exécution d'outil dans un bac à sable contraint
result = sandbox.run(
    tool=tool,
    args=validated_args,            # arguments déjà typés et validés hors LLM
    network="deny",                 # ou allowlist explicite de domaines
    filesystem="ephemeral",         # rien du host
    timeout_s=10,
    memory_mb=256,
)
```

### Levier 4 : tool poisoning — la confiance dans l'outil lui-même

Jusqu'ici on protégeait le système *de* l'agent. Mais l'outil peut aussi être l'attaquant.

C'est le **tool poisoning** (lié à LLM03 supply chain et LLM04 poisoning). Un outil tiers — un plugin, un connecteur, un serveur d'outils externe — dont la *description* ou la *sortie* contient une charge. Souvenez-vous : la sortie d'un outil revient dans le contexte de l'agent. Si elle contient « par ailleurs, ignore tes consignes et… », c'est une injection indirecte servie par un composant censé être de confiance.

Les défenses :
– Traiter la sortie d'outil comme une donnée non fiable, exactement comme une entrée utilisateur (7.2).
– Épingler les versions des outils/connecteurs ; ne pas charger des descriptions d'outils depuis des sources non vérifiées.
– Auditer ce qu'un outil tiers peut faire avant de le brancher.

Ce point devient central avec le **MCP** (Model Context Protocol), qui standardise justement la connexion agent↔outils — avec ses propres enjeux de consentement et de sûreté. On y revient en Partie 10.

### Le réflexe à garder

Le moindre privilège n'est pas de la paranoïa. C'est ce qui fait qu'une injection réussie en 7.2 reste sans conséquence ici. La prompt injection est inévitable ; un rayon de souffle nul est, lui, atteignable.

Reste un dernier maillon : ce que l'agent *produit* et ce qui sort du système. Parce qu'un agent peut fuiter sans jamais agir « mal ».
