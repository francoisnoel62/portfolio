## Fragilité des boucles tool-use

ReAct naïf est une boucle aveugle. On l'a vu en 5.1. Mais le problème prend une forme encore plus concrète quand on regarde ce qui se passe autour des appels d'outils.

Les outils sont le bras armé de l'agent. C'est là qu'il agit sur le monde réel — des API appelées, des bases de données modifiées, des messages envoyés. Et c'est précisément là que les boucles se brisent de la manière la plus spectaculaire.

J'ai identifié trois modes de défaillance. Chacun a des causes distinctes. Chacun nécessite un remède différent.

### Premier mode : la boucle infinie

C'est le plus visible — donc le plus facile à diagnostiquer.

L'agent est bloqué sur une tâche. L'outil retourne une erreur, un résultat vide, ou quelque chose que le modèle n'interprète pas correctement. L'agent raisonne, choisit de réessayer. Même action. Même résultat. Même raisonnement. Boucle.

J'ai vu un agent de gestion des tickets support boucler pendant 23 itérations sur une tâche simple : mettre à jour le statut d'un ticket. L'API de ticketing était en timeout intermittent. L'agent ne le savait pas. Il continuait à appeler `update_ticket_status()` avec la même conviction à chaque fois — jusqu'à ce qu'on coupe manuellement.

```python
# ❌ Boucle infinie — aucun mécanisme d'arrêt
def agent_loop(task, tools, llm):
    context = [{"role": "user", "content": task}]

    while True:  # Pas de limite d'itérations
        response = llm.generate(context)
        action = parse_action(response)

        if action.type == "finish":
            return action.output

        result = tools[action.name](**action.args)
        # Si l'outil échoue, on continue quand même
        context.append({"role": "tool", "content": str(result)})
        # La boucle recommence — sans jamais s'arrêter

# ✅ Limite d'itérations + gestion explicite des erreurs d'outil
def agent_loop(task, tools, llm, max_iterations=10):
    context = [{"role": "user", "content": task}]
    iterations = 0

    while iterations < max_iterations:
        iterations += 1
        response = llm.generate(context)
        action = parse_action(response)

        if action.type == "finish":
            return action.output

        try:
            result = tools[action.name](**action.args)
            context.append({"role": "tool", "content": str(result)})
        except ToolError as e:
            # L'erreur devient une observation — l'agent peut s'adapter
            context.append({"role": "tool", "content": f"Error: {str(e)}"})

    raise MaxIterationsExceeded(f"Agent bloqué après {max_iterations} itérations")
```

La différence ? L'agent ne peut plus boucler indéfiniment. Et les erreurs d'outils deviennent des informations que le modèle peut exploiter — pas des trous noirs qui paralysent la boucle.

### Deuxième mode : les actions contradictoires

C'est plus insidieux. Et j'ai mis des semaines à le comprendre.

Dans une longue boucle, le modèle peut prendre des décisions qui s'annulent mutuellement. Il crée un objet à l'étape 3, oublie qu'il l'a créé à l'étape 9, et le recrée — en écrasant parfois des modifications faites entre-temps.

**Exemple concret :**
J'ai déployé un agent d'orchestration RH chargé de synchroniser des fiches employés entre deux systèmes. À l'étape 4, il crée un enregistrement dans le système B. À l'étape 11, estimant que la synchronisation a échoué (parce qu'une requête de vérification a retourné un résultat ambigu), il recrée le même enregistrement — en écrasant des modifications saisies manuellement par un opérateur.

Le phénomène "lost in the middle" qu'on a vu en partie 3 s'applique aussi ici : quand la boucle s'allonge, le modèle ne "voit" plus clairement ce qu'il a exécuté dix étapes plus tôt. Il raisonne sur ce qui est proche. Le passé lointain s'efface.

La solution complète est dans la planification structurée — on y revient en 5.4 et 5.5. Mais un garde-fou minimal existe : tracer les actions déjà exécutées et refuser les doublons exacts.

```python
# ✅ Déduplication des actions par signature
class AgentLoopWithDedup:
    def __init__(self):
        self.executed_actions = set()

    def should_execute(self, action):
        # Même outil + mêmes arguments = action déjà faite
        signature = (action.name, str(sorted(action.args.items())))
        if signature in self.executed_actions:
            return False, "Action déjà exécutée — skip"
        return True, None

    def execute(self, action, tools):
        can_execute, reason = self.should_execute(action)
        if not can_execute:
            return f"Skipped: {reason}"

        result = tools[action.name](**action.args)
        signature = (action.name, str(sorted(action.args.items())))
        self.executed_actions.add(signature)
        return result
```

Ce n'est pas une solution complète. Mais c'est un filet minimal qui bloque les duplications exactes — les plus fréquentes en production.

### Troisième mode : la dépendance à l'ordre

Les outils ne sont pas commutatifs. L'ordre des appels change le résultat.

Créer un utilisateur puis lui assigner un rôle : ça marche. Assigner un rôle à un utilisateur qui n'existe pas encore : ça échoue. Et dans une boucle ReAct naïve, rien ne garantit que les dépendances entre actions sont respectées — surtout quand la boucle est relancée après une erreur intermédiaire.

**Exemple concret :**
Un agent de provisioning cloud. Mission : créer une infrastructure complète — VPC, sous-réseaux, machines virtuelles, règles de sécurité. Une dizaine d'appels d'outils, avec des dépendances strictes : impossible de créer une VM avant que son sous-réseau existe.

L'agent, relancé après une erreur à mi-parcours, a recommencé certaines étapes sans tenir compte de ce qui était déjà en place. Tentatives de création de ressources déjà existantes, erreurs en cascade, infrastructure dans un état incohérent. Tout ça parce que l'agent n'avait pas de représentation explicite de ce qui était "fait" et de ce qui restait à faire.

La gestion complète des dépendances nécessite de la planification. Mais deux pratiques réduisent les risques à court terme.

Les outils doivent être idempotents. Créer une ressource qui existe déjà ne doit pas lever une erreur fatale — ça doit retourner la ressource existante. Mettre à jour un enregistrement avec les mêmes valeurs doit être sans effet. C'est une convention d'API, mais elle change tout pour un agent qui peut répéter ou re-tenter des actions.

Les résultats d'outils doivent être vérifiables. Après chaque appel critique, l'agent doit confirmer que l'état attendu est bien là. Pas juste "l'outil a retourné 200". Mais "la ressource existe avec les bons attributs".

```python
# ✅ Outil idempotent avec vérification post-action
def create_user_if_not_exists(user_id, email, role):
    existing = db.find_user(user_id)
    if existing:
        return existing  # Idempotent — pas d'erreur, pas de doublon

    user = db.create_user(user_id=user_id, email=email, role=role)

    # Vérification explicite — on ne fait pas confiance au retour de l'outil
    assert db.find_user(user_id) is not None, f"User {user_id} not found after creation"
    return user
```

### Ce que ça change en pratique

Trois modes. Trois types de garde-fous.

Pour les boucles infinies : `max_iterations`, gestion explicite des exceptions d'outils, exception métier claire en cas de dépassement.

Pour les actions contradictoires : déduplication par signature d'action, historique des actions consultable à chaque étape.

Pour les dépendances d'ordre : outils idempotents, vérification post-action, et planification structurée — qu'on aborde en 5.4.

Ces mécanismes n'éliminent pas tous les problèmes. Mais ils suppriment les modes de défaillance les plus courants — ceux qui génèrent des incidents à 2h du matin.

Il reste un problème plus difficile à corriger, parce qu'il est invisible de l'extérieur : les erreurs ne surviennent pas uniquement dans les outils — elles se propagent d'une étape à l'autre dans le raisonnement lui-même, et l'agent ne le sait pas. C'est l'effet domino qu'on explore dans la section 5.3.
