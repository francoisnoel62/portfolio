## Fenêtre ≠ mémoire

Il y a une confusion que je vois dans pratiquement tous les projets d'agents que j'ai accompagnés.

Les développeurs utilisent la fenêtre de contexte comme si c'était une mémoire. Ils y accumulent l'historique de conversation, les préférences utilisateur, les décisions précédentes. Et pendant une session, ça marche. L'agent semble se souvenir. Il fait référence à ce qui a été dit. Il paraît cohérent.

L'illusion est parfaite.

Jusqu'à ce que la session se ferme.

Au prochain appel, le contexte est vide. L'agent repart de zéro. Il ne sait plus qui est l'utilisateur. Il ne se souvient pas des décisions prises. Il repose les mêmes questions. Et l'utilisateur, lui, ne comprend pas pourquoi il doit tout réexpliquer à chaque fois.

C'est l'une des sources les plus fréquentes de friction en production. Pas un bug. Pas une erreur de modèle. Une confusion architecturale.

**Le problème :** la fenêtre de contexte a trois limites fondamentales que la mémoire n'a pas.

Elle est **temporaire**. Elle existe le temps d'une requête, d'une session. Quand l'appel se termine, tout s'efface. Il n'y a pas de persistance native. Ce que le modèle "savait" dans l'échange précédent, il ne le sait plus maintenant.

Elle est **limitée**. Même avec 128K tokens, on ne peut pas stocker l'historique complet d'un utilisateur qui interagit avec le système depuis six mois. Et comme on l'a vu en 3.1, remplir la fenêtre ne signifie pas que le modèle traite tout avec la même attention.

Elle est **non structurée**. Tout est du texte. On ne peut pas faire de requêtes sur le contexte. On ne peut pas filtrer, trier, agrégrer. Si on cherche "toutes les décisions prises concernant le client Dupont", il faut parcourir la totalité de l'historique — ce qui n'est ni fiable ni scalable.

**Exemple concret :**

J'ai travaillé sur un agent de service client pour une plateforme e-commerce. L'équipe avait fait quelque chose de raisonnable en apparence : à chaque requête, elle injectait les 10 derniers échanges de l'utilisateur dans le contexte. L'agent semblait "se souvenir" des interactions récentes.

Le problème : un utilisateur qui revenait après deux semaines voyait l'agent lui reposer des questions déjà posées. Et un utilisateur avec un historique dense — beaucoup d'achats, beaucoup de tickets de support — consommait une quantité disproportionnée de tokens rien que pour reconstituer le contexte, avant même d'avoir posé sa question.

L'équipe avait confondu deux choses : l'historique récent (utile dans la session courante) et la mémoire long terme (ce que l'agent doit savoir sur cet utilisateur, en permanence).

Ce qui devait rester dans la fenêtre : les derniers échanges de la session en cours.
Ce qui devait être externalisé : les préférences utilisateur, l'historique des commandes, les décisions prises lors de tickets précédents.

```python
# ❌ Tout dans le contexte — illusion de mémoire
def handle_query(user_id: str, query: str, session_history: list) -> str:
    # On injecte l'historique complet dans le contexte
    all_history = db.get_all_interactions(user_id)  # Potentiellement des centaines d'échanges
    context = build_context(
        system_prompt=SYSTEM_PROMPT,
        history=all_history,  # Tout, sans discernement
        query=query
    )
    return llm.generate(context)


# ✅ Séparation fenêtre / mémoire
def handle_query(user_id: str, query: str, session_history: list) -> str:
    # Mémoire long terme : récupérée depuis le store externe
    user_profile = memory_store.get_profile(user_id)       # Préférences, historique structuré
    relevant_facts = memory_store.search(user_id, query)   # Faits pertinents pour cette requête

    # Fenêtre : uniquement la session courante
    context = build_context(
        system_prompt=SYSTEM_PROMPT,
        user_profile=user_profile,      # Compact, structuré
        relevant_facts=relevant_facts,  # Ciblé, retrieval dynamique
        session_history=session_history, # Cette session uniquement
        query=query
    )
    return llm.generate(context)
```

La différence n'est pas anecdotique. Dans le premier cas, on alloue des tokens à des informations qu'on espère pertinentes. Dans le second, on alloue des tokens à des informations qu'on sait pertinentes.

Ce qui doit être externalisé suit une logique simple : tout ce qui doit survivre à la fin d'une session. Les préférences utilisateur. Les décisions prises dans des échanges précédents. Le state structuré de l'agent (où en est-on dans un workflow multi-étapes ?). Les faits appris qui doivent influencer les prochaines interactions.

Ce qui reste dans la fenêtre : la session en cours. L'échange immédiat. Le contexte dynamique récupéré par retrieval.

[SCHÉMA 3.3] Fenêtre de contexte vs mémoire externalisée : ce qui disparaît vs ce qui persiste

La fenêtre de contexte est un espace de travail, pas un disque dur.

La confondre avec une mémoire, c'est construire un agent qui ne peut fonctionner que tant qu'on ne ferme pas la page. C'est acceptable en démo. En production, avec des utilisateurs réels qui reviennent, c'est une régression à chaque nouvelle session.

La Partie 4 reviendra en détail sur l'architecture mémoire complète. Mais avant ça, il reste un problème dans la gestion du contexte dynamique : le retrieval. Parce que décider quoi mettre dans la fenêtre à partir d'un store externe — ça ne s'improvise pas non plus.
