## Mémoire court terme vs long terme

On a posé le problème en 4.1 : pas de mémoire native, pas de continuité sans architecture explicite. Maintenant, construisons cette architecture.

La première décision à prendre n'est pas *où* stocker. C'est *quoi* stocker et *combien de temps*. Parce que tout ce qu'on appelle "mémoire" n'est pas du même type. Traiter l'ensemble de manière uniforme est l'erreur la plus courante que je vois sur le terrain — et elle coûte cher dans les deux directions : soit l'agent oublie ce qu'il devrait retenir, soit il accumule un stock de souvenirs qui ne servent à rien.

Il y a deux registres. Deux logiques différentes. Deux types de stores. Et les confondre casse l'architecture avant même qu'on l'ait construite.

### La mémoire court terme : ce qui sert maintenant

La mémoire court terme, c'est ce qui est utile dans la session courante. Pour la tâche en cours. Ici et maintenant.

Elle contient l'historique des échanges de la conversation active, les résultats intermédiaires des outils utilisés pendant la session, et les données contextuelles spécifiques à la requête. Sa durée de vie, c'est la session. Quand la session se termine, cette mémoire n'a plus vocation à exister — et c'est délibéré.

Prenons un exemple concret : j'utilise mon agent pour analyser un contrat spécifique. Les détails de ce contrat — clauses, parties prenantes, dates de validité — sont critiques pour cette session. Dans six mois, si je reviens avec une requête différente, ces détails n'ont probablement aucune valeur. Les injecter dans le contexte serait du bruit. Les stocker indéfiniment serait du coût sans bénéfice.

La mémoire court terme vit dans le contexte de session, en mémoire vive ou dans un cache éphémère avec un TTL court. Elle est rapide d'accès parce qu'elle est proche. Volatile par design.

### La mémoire long terme : ce qui a de la valeur au-delà

La mémoire long terme, c'est ce qui mérite d'être retrouvé demain, dans deux semaines, dans trois mois.

Elle contient les profils utilisateurs, les préférences, les décisions passées significatives, les patterns récurrents observés sur cet utilisateur ou ce contexte. Elle est externe au LLM — stockée dans une base de données, un vector store, ou les deux selon le type d'information et la nature de la requête.

La différence avec la mémoire court terme n'est pas seulement technique. Elle est conceptuelle. La mémoire long terme répond à une question précise : "Qu'est-ce que cet agent devrait savoir sur cet utilisateur à la prochaine session ?" La réponse à cette question est exactement ce qui mérite d'être persisté. Pas plus.

**Exemple concret :**

J'ai déployé un agent SAV pour une boutique e-commerce. Dans les premières semaines, tout fonctionnait correctement. Puis des anomalies ont commencé à apparaître.

L'agent rappelait à un client qu'il avait eu "un problème de livraison récent" — alors que ce problème datait de six mois. La boutique avait changé de transporteur depuis. L'information était factuellement correcte mais totalement hors-propos. Le client était perplexe. Et le transporteur incriminé n'était plus actif depuis des mois.

Le diagnostic était simple : on stockait tout en long terme, sans critère de durée de vie. Le suivi d'une commande spécifique, le contexte d'un litige clôturé, la référence d'un produit retourné — des informations éphémères traitées comme des connaissances durables.

L'adresse de livraison d'un client ? Long terme — stable, utile à chaque commande. Le détail d'une réclamation résolue six mois plus tôt ? Court terme, avec un TTL d'une semaine après clôture.

### Le critère qui tranche : la pertinence future probable

Pour chaque élément d'information, une seule question : "Est-ce que j'aurai besoin de ça lors d'une prochaine session, dans un contexte similaire ?" Si oui, long terme. Sinon, court terme ou rien.

Ce n'est pas un critère d'importance immédiate. C'est un critère de **durabilité de la pertinence**.

```python
# ❌ Architecture sans distinction — tout en long terme
def save_exchange(user_id: str, exchange: dict) -> None:
    long_term_store.save(user_id, exchange)  # On stocke tout, sans discernement


# ✅ Architecture à deux couches avec consolidation sélective
class AgentMemory:
    def __init__(self, user_id: str):
        self.user_id = user_id
        self.short_term = []                        # Session courante, volatile
        self.long_term = LongTermStore(user_id)     # Persistant, externe

    def add_exchange(self, exchange: dict, consolidate: bool = False) -> None:
        # Court terme : toujours — c'est la mémoire de travail de la session
        self.short_term.append(exchange)
        # Long terme : seulement si l'information a une valeur future prouvable
        if consolidate:
            self.long_term.save(exchange)

    def build_context(self, query: str) -> list:
        # Ce que l'agent sait durablement sur cet utilisateur
        durable_context = self.long_term.get_relevant(query, top_k=3)
        # Les échanges récents de la session en cours
        recent_context = self.short_term[-6:]
        return durable_context + recent_context
```

La décision de consolider (`consolidate=True`) se prend en amont, pas dans la boucle d'inférence. C'est une décision architecturale : quels types d'échanges méritent de passer en long terme ? Les préférences explicites de l'utilisateur, oui. Un résultat intermédiaire de calcul spécifique à cette session, non.

### Le piège de tout garder

Il existe une tentation naturelle : stocker tout en long terme "au cas où". Je comprends l'impulsion — on ne sait jamais ce qui sera utile plus tard.

Mais cette approche a des coûts qui se révèlent vite. Le long-term store grossit sans contrôle. Le retrieval devient plus lent, plus coûteux, moins précis — exactement le syndrome du top-k trop élevé qu'on a vu en section 3.4. Et les anciennes informations bruitent les nouvelles : un agent qui "se souvient" de tout finit par se souvenir de choses contradictoires ou périmées, sans signal clair sur ce qui est encore valide.

Une mémoire long terme efficace est une mémoire éditée. Ce qui entre a passé un critère. Ce qui n'est plus pertinent peut être supprimé ou archivé avec un TTL.

Les trade-offs sont réels : la mémoire court terme est rapide mais éphémère. La mémoire long terme est persistante mais coûteuse à récupérer et à maintenir. Il n'y a pas d'architecture parfaite — il y a une architecture calibrée sur ce que le système doit réellement retenir.

[SCHÉMA 4.2] Architecture mémoire à deux couches : court terme (session, volatile) vs long terme (store persistant, requête sélective)

La distinction court terme / long terme est le premier axe de l'architecture mémoire. Mais cette architecture a un autre niveau de confusion fréquente : l'historique de conversation n'est pas l'état de l'agent. Et les traiter de la même façon, c'est la prochaine erreur à éviter.
