## La grille de gouvernance

On a vu le cadre légal (8.2), le cadre de processus (8.3), et comment les traduire en code (8.4). Reste à en faire un outil de décision tenant sur une page.

Parce que la gouvernance échoue presque toujours pour la même raison : elle vit dans des têtes différentes. Le juridique connaît les obligations, l'ingénieur connaît l'architecture, personne ne fait le lien, et au moment du déploiement on découvre le trou. La grille ferme ce trou.

### Une ligne par agent

[TABLEAU 8.3] Grille de gouvernance : une ligne par agent

| Colonne | Question | Renvoi |
|---|---|---|
| **Agent** | Quel système ? | — |
| **Usage réel** | Que décide-t-il, pour qui ? | 8.2 |
| **Niveau de risque** | Inacceptable / haut / limité / minimal | 8.2 |
| **Obligations** | Lesquelles s'appliquent ? | 8.2 (Art. 9–15) |
| **Contrôle architectural** | Quelle brique y répond ? | Parties 2, 6, 7, 8.4 |
| **Propriétaire** | Qui en répond ? | GOVERN (8.3) |
| **Statut** | Conforme / écart / bloqué | — |

L'intérêt de cette grille n'est pas bureaucratique. C'est un **outil de priorisation**. Elle rend visible, d'un coup d'œil, où sont les écarts : un agent à haut risque sans propriétaire, une obligation sans contrôle en face, une supervision humaine cochée mais non implémentée.

**Exemple concret :**
Pour l'agent de présélection RH, la ligne donnait : usage = tri de candidats ; risque = haut ; obligations = Art. 9/10/12/14/15 ; contrôles = threat model (7.5), `ComplianceRecord` append-only (8.4), gating recruteur (Partie 2), eval baseline (6.2) ; propriétaire = responsable recrutement ; statut = conforme. Une ligne, et tout le monde — juridique, technique, métier — voyait la même réalité. La ligne d'à côté, un agent de scoring sans propriétaire, est ressortie en rouge immédiatement.

### Le point que tout le monde oublie : l'ownership

Si vous ne retenez qu'une chose de cette partie, retenez celle-ci : **chaque agent doit avoir un propriétaire qui répond de ses décisions.**

Pas l'équipe. Pas « l'IA ». Une personne ou un rôle, nommé, avec l'autorité de suspendre l'agent. C'est le cœur de la fonction GOVERN du NIST, et c'est ce qui manque le plus souvent en pratique.

Sans propriétaire :
- le drift (6.4) n'est corrigé par personne ;
- les cas limites s'accumulent sans arbitre ;
- en cas d'incident, la question « qui décide d'arrêter ? » reste sans réponse — au pire moment.

Un agent sans propriétaire n'est pas un agent autonome. C'est un agent abandonné.

Et le propriétaire n'agit pas à mains nues. Il s'appuie sur ce qu'on construira en Partie 9 : un runbook qui dit quoi faire quand l'agent dérape, une astreinte capable de l'arrêter, un chemin de rollback. La grille *nomme* le responsable ; les opérations lui donnent les *moyens* d'agir. Nommer un propriétaire sans lui donner de bouton d'arrêt, c'est désigner un coupable, pas un pilote. Les deux vont ensemble.

### Deux artefacts qui prouvent

La grille s'appuie sur deux documents légers mais décisifs.

- **La fiche système (system card).** Une page par agent : ce qu'il fait, ses données, son modèle et sa version, ses limites connues, ses métriques de référence (les baselines de 6.2), son propriétaire. C'est la base de la documentation technique de l'Art. 11, et c'est aussi ce qu'on tend à un client ou un auditeur.
- **La piste d'audit.** Le `ComplianceRecord` append-only de 8.4. La fiche dit ce que le système *est censé* faire ; la piste d'audit prouve ce qu'il *a fait*. Les deux ensemble répondent à n'importe quelle question d'un régulateur.

### Le go/no-go avant déploiement

Dernière brique : une porte. Avant qu'un agent à haut risque parte en production, on passe la grille en revue. Si une obligation n'a pas de contrôle en face, ou si l'agent n'a pas de propriétaire, c'est **no-go**. Pas un avis. Une règle.

```python
# ✅ Porte de déploiement : la conformité comme condition, pas comme option
def deployment_gate(agent: AgentSpec) -> Decision:
    if agent.risk_tier == "unacceptable":
        return block("usage interdit par l'EU AI Act")
    if agent.risk_tier == "high":
        missing = agent.obligations - agent.implemented_controls
        if missing:
            return block(f"obligations sans contrôle : {missing}")
        if agent.owner is None:
            return block("aucun propriétaire désigné (GOVERN)")
    return approve(agent)
```

Cette porte n'est pas là pour ralentir. Elle est là pour éviter le scénario de 8.1 : un agent techniquement excellent, non déployable parce que personne n'a pensé la gouvernance avant la veille de la mise en prod.

### Le livrable de la Partie 8

Vous repartez avec quatre choses : la **classification de risque** (8.2), le mapping **obligation → contrôle architectural** (8.4), la **grille de gouvernance** avec propriétaire nommé, et une **porte de déploiement** go/no-go. La conformité cesse d'être un classeur. Elle devient une propriété du système, prouvable et possédée.

Mise en garde finale, déjà dite mais essentielle : ce paysage réglementaire évolue vite. Les principes — classer, contrôler, tracer, posséder — sont stables. Les échéances et les détails, non. Vérifiez le statut courant de l'EU AI Act au moment où vous déployez.

Gouverner un agent, c'est bien. Encore faut-il le faire **tourner** : retries, idempotence, files, rollback, astreinte. La gouvernance reste théorique tant que les opérations ne suivent pas. C'est la Partie 9.
