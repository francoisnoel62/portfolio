## Choisir sa couche d'outillage : la décision de framework

On a maintenant quatre couches distinctes à disposition :

- **MCP** : protocole de connexion agent↔outil
- **A2A** : protocole de communication agent↔agent
- **Agents SDK** (OpenAI ou équivalent) : runtime pour orchestrer plusieurs agents avec guardrails, handoffs, sessions
- **Sandbox** (E2B, Docker) : isolation d'exécution pour le code généré ou les outils risqués

La question n'est pas "laquelle utiliser ?". C'est "laquelle *ajoute de la valeur* à mon cas ?"

### La règle qui n'a pas changé

En 1.5, j'ai posé la règle centrale du livre : **n'ajouter de l'agent que quand la complexité de décision dépasse la complexité d'exécution**. La même règle s'applique au choix de couche d'outillage.

Chaque couche ajoute de l'indirection. L'indirection a un coût : latence, surface de débogage, dépendances supplémentaires, configuration à maintenir. Elle n'est justifiée que si elle apporte une valeur que le code direct ne peut pas donner.

### La matrice de décision

[TABLEAU 10.1] Matrice de choix d'outillage — cas × couche recommandée

| Cas | Ce dont tu as besoin | Recommandation |
|-----|---------------------|----------------|
| Un agent + 2–3 tools stables, stack homogène | Wrapper function simple | Rien de plus. Pas de MCP, pas de SDK. |
| Un agent + tools variés, intégrations changeantes | Contrat standard entre agent et tools | MCP |
| Plusieurs agents spécialisés, même codebase | Routing et délégation | Agents SDK (handoffs) |
| Agents multi-stack, multi-éditeur, multi-entreprise | Interop sans couplage | A2A |
| Agent qui génère ou exécute du code | Isolation de l'exécution | Sandbox (E2B, Docker) |
| Agent en production avec SLO, incidents, rollback | Observabilité structurée | OTel + backend de tracing |

Ce tableau simplifie. Dans la pratique, les colonnes se cumulent : un agent multi-stack qui exécute du code a besoin de A2A *et* d'un sandbox. Mais le principe reste : commencer par la couche la plus simple, et n'ajouter la suivante que quand on a un problème concret qu'elle résout.

### Quand MCP suffit

MCP est le bon choix quand le problème est l'intégration : tu as un agent et plusieurs tools, chacun avec son format d'API. MCP standardise le contrat — plus de wrappers custom.

Il ne faut *pas* MCP si tu as un seul outil stable avec une API bien définie. Un wrapper Python direct est plus simple, plus rapide à déboguer, et ne dépend d'aucun serveur MCP.

### Quand A2A s'impose

A2A est nécessaire quand plusieurs agents doivent collaborer sans se connaître en détail — parce qu'ils viennent de stacks différentes, d'équipes différentes, ou de fournisseurs différents.

Si tous tes agents sont dans le même codebase Python, un handoff SDK est plus simple : pas de sérialisation réseau, pas d'Agent Card à maintenir, pas de transport HTTP. A2A est fait pour l'interopérabilité *across* systèmes, pas pour l'orchestration *within* un système.

### Quand un SDK maison est préférable

Les Agents SDKs (OpenAI, LangGraph, Pydantic AI…) imposent des abstractions. Ces abstractions sont utiles — guardrails déclaratifs, tracing intégré, sessions — mais elles ont un coût : elles décident à ta place de certains comportements.

Si ton cas est simple (un agent, pas de délégation, lifecycle trivial), un SDK ajoute de la complexité inutile. Si ton cas est très spécifique (orchestration durable multi-step avec saga pattern, lien 9.3), le SDK ne couvrira pas tous tes besoins et tu finiras quand même à écrire du code custom *par-dessus* les abstractions du SDK — ce qui est le pire des deux mondes.

La règle : un SDK de référence est excellent pour démarrer et pour les patterns courants. Dès que tu te bats contre ses abstractions, c'est le signal pour en sortir.

```python
# ❌ Forcer un pattern custom dans un SDK qui ne le supporte pas
class HackedHandoff(Handoff):
    # 200 lignes pour contourner une limitation du SDK
    # On perd les guarantees du SDK et on hérite de ses bugs
    ...

# ✅ Implémenter directement ce dont on a besoin
async def route_to_specialized_agent(context, task_type):
    # 20 lignes de logique claire, sans dépendance SDK
    agent = agent_registry[task_type]
    return await agent.run(context)
```

### Le principe framework-agnostic du livre

Ce livre n'a pas été écrit pour un framework. Les patterns qu'il décrit — mémoire externalisée (Partie 4), planification structurée (Partie 5), evaluation dès le jour zéro (Partie 6) — s'implémentent dans n'importe quelle stack.

MCP, A2A, l'Agents SDK : ce sont des implémentations de ces patterns, pas les patterns eux-mêmes. Comprendre *pourquoi* on a besoin d'un contrat entre agent et outil (MCP) vaut plus que de savoir configurer un serveur MCP. Comprendre *pourquoi* les guardrails existent (validation à la frontière système/données) vaut plus que de connaître l'API `InputGuardrail`.

Les frameworks changent. Les problèmes, non.

**Exemple concret :**
Hotelix (Partie 11) : un agent de traitement de tickets hôteliers qui semblait justifier un Agents SDK complet — routage entre agents spécialisés (accueil, maintenance, facturation), guardrails sur les réponses, sessions persistantes. En pratique, après analyse, 90 % des tickets suivaient un workflow déterministe. Le bon choix était un workflow conditionnel simple avec une seule invocation LLM pour les cas ambigus. Pas de SDK, pas d'A2A, pas de multi-agent. Un workflow. L'agent était une mauvaise idée — et la couche d'outillage était la mauvaise question à se poser en premier.
